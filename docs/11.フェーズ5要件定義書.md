# 【フェーズ5要件定義書 / タスク名】
お前の任務は、**フェーズ1〜4で整えた基盤を統合し、能力探索・選定・活用・実行判断を一貫して担うAI参謀を完成させること** である。

このフェーズは最終段階である。  
ここで作るのは、単なる検索器でも、単なるルーターでも、単なる実行者でもない。  
**ユーザーの目的に対して、使うべき能力を見つけ、選び、根拠付きで提案し、実行フローへ接続できる統合参謀** を定義する。

---

# 【このフェーズの目的】
このフェーズの目的は次の5つだけである。

1. **情報基盤・判断基盤・知識活用基盤・実行基盤を一つの参謀ロジックへ統合する**
2. **ユーザーの依頼に対して、使うべき機能・MCP・API・OSS・既存エージェント・作業フローを選定できるようにする**
3. **選定理由を、根拠付きで説明できるようにする**
4. **必要なときは最新情報を再確認し、古い知識だけで断定しないようにする**
5. **実行基盤へ橋渡しし、最短かつ高品質で作業を前進させる状態にする**

---

# 【このフェーズの位置づけ】
- フェーズ1：情報を集める
- フェーズ2：何を使うか判断できるようにする
- フェーズ3：知識として引けるようにする
- フェーズ4：実作業で最大活用できるようにする
- **フェーズ5：それらを統合し、AI参謀として完成させる**

---

# 【入力】
このフェーズで前提とする入力は以下である。

- フェーズ1の情報資産
- フェーズ2の比較結果・採用状態・優先順位
- フェーズ3の知識活用ルール・再確認ルール・信頼区分
- フェーズ4の実行フロー・ツール投入方針・品質保証ルール
- GitHub上のコード、履歴、比較対象、参考実装
- 実行可能な公式機能、MCP、API、OSS、既存エージェント
- 継続的に更新される公式情報と公開情報

---

# 【このフェーズで扱う対象】
## 1. 依頼理解
- ユーザーが何を達成したいか
- 今回どこまでやるべきか
- 何をまだやらないか

## 2. 能力選定
- 使うべき機能
- 使うべきツール
- 使うべき既存資産
- 必要な再確認
- 最適な作業フロー

## 3. 実行接続
- 実行基盤への橋渡し
- 作業開始条件
- 完了条件
- 検証条件
- GitHub反映条件

## 4. 継続改善
- 新機能の取り込み
- 既存判断の更新
- 再評価条件
- 使うべき能力の見直し

---

# 【最重要原則】
## 原則1｜参謀は判断に責任を持つ
参謀の役割は、候補を並べることではない。  
**何を使うべきかを決めること** にある。

## 原則2｜使える能力は積極的に使う
使えると判断した機能・ツール・既存資産は、実行可能な形で結びつける。  
能力を知っているだけで終わるな。

## 原則3｜理由を必ず持つ
選定結果には必ず理由を持たせる。  
感覚で選ぶな。

## 原則4｜知識だけで断定しない
鮮度が重要な内容は、必要に応じて再確認する。  
保存知識だけで現在の事実を断定するな。

## 原則5｜最終目的から逆算する
参謀は、機能中心ではなく目的中心で判断する。  
「何があるか」より「何が最適か」を優先する。

## 原則6｜実行できる提案だけを出す
理想論だけを返すな。  
実際に実行基盤へ接続できる提案だけを出すこと。

---

# 【このフェーズで実現すべき状態】
フェーズ5完了時点で、最低限以下を満たせ。

1. **依頼を受けて、必要な能力を選定できる**
2. **機能・MCP・API・OSS・既存エージェント・作業フローを組み合わせて提案できる**
3. **提案理由を説明できる**
4. **鮮度確認が必要な内容を見抜ける**
5. **実行基盤へそのまま渡せる形にできる**
6. **新しい能力が出たときに、再評価して取り込める**
7. **効率と品質の両方を最大化する判断ができる**

---

# 【このフェーズで必要なコア機能】
## 機能A｜依頼分類
依頼を最低限、以下のどれかに分けられること。
- 調査中心
- 比較中心
- 実装中心
- 修正中心
- 設計中心
- 運用改善中心

## 機能B｜能力選定
依頼に対して、最低限以下を選べること。
- 使うべき公式機能
- 使うべきMCP / API
- 使うべきOSS / 既存エージェント
- 使うべき実行フロー
- 再確認が必要な情報源

## 機能C｜理由生成
選定結果に対して、最低限以下を示せること。
- なぜそれを使うか
- なぜ他を使わないか
- どこに効くか
- 何に注意すべきか

## 機能D｜実行橋渡し
最低限、以下を実行基盤へ渡せること。
- 今回の目的
- 今回使う能力
- 今回の制約
- 完了条件
- 検証条件

## 機能E｜更新吸収
新しいツールや新機能を見つけたときに、
- 再探索
- 再評価
- 既存判断との差分確認
- 必要なら採用更新
ができること。

---

# 【このフェーズで定義すべき判断出力】
最終的な参謀は、最低限以下を出力できること。

1. **今回の目的**
2. **今回使うべき能力**
3. **今回使わない能力**
4. **その理由**
5. **必要な再確認**
6. **推奨作業フロー**
7. **完了条件**
8. **検証条件**

---

# 【このフェーズで定義すべき更新ルール】
## ルール1｜新機能発見時
新機能を見つけたら、
- 何に効くか
- 今の構成を改善するか
- 既存候補を置き換えるか
を確認する。

## ルール2｜既存判断の陳腐化検知
以下の場合は、既存判断を見直す。
- 公式仕様変更
- 提供状態変更
- 主要OSSの更新
- より良い代替候補の出現

## ルール3｜再評価
再評価対象は、
- 比較し直す
- 優先順位を見直す
- 実行フローへの影響を確認する

---

# 【このフェーズでまだやらないこと】
- 次の巨大な新フェーズを増やすこと
- 全能力の完全自動採用
- 人間確認ゼロの無制限自律運用
- 監督なしの全面自己進化

---

# 【完了条件】
以下を満たしたときのみ、フェーズ5完了とみなす。

1. 依頼を受けて必要能力を選べる
2. 選定理由を説明できる
3. 必要な再確認を判断できる
4. 実行基盤へ渡せる
5. 新しい能力を取り込める
6. 効率と品質を最大化する統合参謀になっている

---

# 【絶対ルール】
- 候補を並べるだけで終わるな
- 理由のない判断をするな
- 古い知識だけで断定するな
- 実行できない提案をするな
- 使える能力を眠らせるな
- 目的より機能を優先するな
- フェーズ1〜4の基盤を無視するな
