# 【フェーズ4要件定義書 / タスク名】
お前の任務は、**フェーズ1〜3で整えた情報基盤・判断基盤・知識活用基盤を、実際の作業で最大効率・最高品質を出すための実行基盤へ変換すること** である。

このフェーズでは、**使えると判断したツールは最大限活用する**。  
補助として遠慮がちに使うのではなく、**実行の主戦力として使う**。  
また、進行中に新しく有効なツールを見つけた場合は、後回しにせず、**その場で評価し、使えるなら即時に作業フローへ組み込む**。

---

# 【このフェーズの目的】
フェーズ4の目的は、次の4つだけである。

1. **実作業をツール中心のフローへ置き換える**
2. **使えるツールを最大限活用して、効率と品質を同時に最大化する**
3. **調査・比較・実装・修正・検証・GitHub反映を一つの標準フローにする**
4. **フェーズ5でAI参謀がそのまま使える実行基盤を作る**

---

# 【このフェーズの位置づけ】
- フェーズ1：情報を集める
- フェーズ2：何を使うか決める
- フェーズ3：後から引ける知識にする
- **フェーズ4：それらを使って実際に最速・高品質で作業する**
- フェーズ5：AI参謀として統合する

---

# 【入力】
このフェーズで前提とする入力は以下である。

- フェーズ1の情報資産
- フェーズ2の評価結果・採用候補・優先順位
- フェーズ3の知識活用ルール・再確認ルール
- GitHub上のコード・履歴・比較対象
- 利用可能な公式機能・MCP・API・OSS・既存エージェント
- フェーズ4中に新しく見つかる有効ツール

---

# 【対象】
このフェーズで扱う対象は次の通り。

## 1. 実行フロー
- 調査
- 比較
- 実装
- 修正
- 検証
- 記録
- GitHub反映

## 2. ツール投入
- 既に使えるツール
- 新しく見つかるツール
- 既存OSS
- 公式機能
- GitHub連携資産

## 3. 品質保証
- 自己検証
- 差分確認
- 完了条件確認
- 未完了報告
- 後方影響確認

---

# 【最重要原則】
## 原則1｜使えるツールは最大限使う
使えると判断したものは、限定的に使うのではなく、**最大限活用する**。  
それがこのプロジェクトの前提である。

## 原則2｜新しい有効ツールは即時投入する
新しい有効ツールを見つけたら、
**役割確認 → 導入可否判断 → 使えるなら即投入**
まで一気に進める。  
「あとで検討」は標準にしない。

## 原則3｜効率と品質を同時に上げる
速いだけでは不十分。  
品質が高いだけでも不十分。  
**効率化と品質向上の両方を最大化する**。

## 原則4｜実行前に入口と出口を固定する
毎回の作業で、最低限以下を先に固定する。
- 今回何をやるか
- どこまでやれば止まるか
- 何をもって完了とするか

## 原則5｜自己検証を必須にする
主要作業には必ず自己検証を含める。
- テスト
- ビルド
- 差分確認
- 出力確認
- 再現確認

## 原則6｜GitHubを実行基盤に接続する
実行結果は、必ず GitHub の
- 差分
- コミット
- 履歴
- 比較対象
- 復帰可能性
へ接続すること。

---

# 【このフェーズで実現すべき状態】
フェーズ4完了時点で、最低限以下を満たせ。

1. **ツール最大活用前提の実行フローが定義されている**
2. **作業単位ごとの標準フローがある**
3. **新規ツールの即時投入ルールがある**
4. **自己検証が組み込まれている**
5. **GitHub反映まで含めて標準化されている**
6. **フェーズ5へ渡せる実行基盤になっている**

---

# 【このフェーズで定義すべき作業単位】
最低限、以下の作業単位を標準化すること。

## 調査
- 既知知識参照
- 必要な公式再確認
- GitHub確認
- 使用ツール選定

## 比較
- 比較対象選定
- 比較基準適用
- 採用 / 保留 / 不採用整理

## 実装
- 対象固定
- 使用ツール固定
- 実装
- 自己検証
- 差分確認

## 修正
- 問題把握
- 修正
- 再検証
- 必要時は撤退とやり直し

## 記録
- 何を使ったか
- 何を変えたか
- どう検証したか
- 未完了は何か
- 次に何をやるか

---

# 【このフェーズで定義すべきツール投入ルール】
## ルール1｜既存採用候補は主戦力
フェーズ2で高優先・最優先になったものは、フェーズ4では原則として活用前提で扱う。

## ルール2｜作業前にツール選定を必ず行う
各作業開始前に、最低限以下を確認する。
- 今回使えるツールは何か
- 今回もっとも効くツールは何か
- 追加で使える新規候補はないか

## ルール3｜新規ツールは即時評価する
新しいツールを見つけたら、最低限以下を確認する。
- 今回の作業に直接効くか
- 既存フローより改善できるか
- すぐ使えるか

## ルール4｜使えるなら即使う
使えると判断したら、その作業フローへ即投入する。

---

# 【このフェーズで定義すべき品質保証】
最低限、以下を必須にすること。

- 自己検証
- 差分確認
- 完了報告
- 未完了報告
- 影響範囲確認
- GitHub反映
- 必要時の復帰可能性

---

# 【このフェーズで定義すべきGitHub接続】
最低限、以下を満たすこと。

- 変更内容が差分で見える
- 1タスク単位で履歴が追える
- 比較対象OSSや参考実装に接続できる
- 正常状態へ戻せる
- 進捗確認ができる

---

# 【このフェーズでまだやらないこと】
以下はフェーズ4ではまだやらない。

- 完成版AI参謀の最終統合
- 完全自律運用の完成
- 人間確認ゼロの全面自動化
- フェーズ5の最終統合ロジック
- 全ツールの完全網羅保証

---

# 【成果物】
このフェーズで必要な成果物は次だけでよい。

1. **実行フロー定義**
2. **ツール最大活用方針**
3. **新規ツール即時投入ルール**
4. **作業単位定義**
5. **品質保証ルール**
6. **GitHub接続方針**
7. **フェーズ5接続メモ**

---

# 【完了条件】
以下を満たしたときのみ、フェーズ4完了とみなす。

1. ツール最大活用が原則として定義されている
2. 新規ツールの即時投入ルールがある
3. 作業単位ごとの標準フローがある
4. 自己検証が必須化されている
5. GitHub接続が組み込まれている
6. フェーズ5へ渡せる実行基盤になっている

---

# 【絶対ルール】
- 使えるツールを遠慮して使うな
- 旧来の手作業中心に戻るな
- 新規ツールを後回し前提にするな
- 効率だけを上げて品質を落とすな
- 品質だけを守って速度を落とすな
- 自己検証なしで完了扱いするな
- GitHubに接続しない実行フローにするな
- フェーズ5の完成物を先取りしない

---

次に行うべきことは、**この要件定義書を正本としたフェーズ4作業指示書の作成** である。
