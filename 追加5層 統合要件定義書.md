# 【追加5層 統合要件定義書 / タスク名】
お前の任務は、**既存の頭脳（依頼分類・能力選定・モデル選定・再確認判断・実行橋渡し）を前提として、その頭脳が手足を自動運用できる5層を優先順位順に実装し、完成した層はその瞬間から後続作業で最大限活用すること** である。

---

# 【前提】
- 既存の頭脳は **再実装しない**
- 今回やるのは **追加5層の構築**
- 各層は **完成後ただちに標準実行フローへ組み込む**
- 後続層の実装は、**先に完成した層を使って進める**
- 目的は **最大効率・最高品質・最小手動操作** である

---

# 【優先順位】
## 1. 実行制御層
AIが **slash command / subagent / hook / 既存ツール / 実行フロー** を自動選択・自動発火・自動接続する層。

## 2. 自動品質ゲート層
実行のたびに **テスト / 差分確認 / 危険変更検知 / 失敗停止** を自動で入れる層。

## 3. 接続・認証・権限自動化層
MCP / API / GitHub / 公式機能を **使える状態まで自動で整える層**。

## 4. 常時監視・自動同期層
公式サイト / GitHub / 重要OSS を継続監視し、**更新・新機能・破壊的変更** を取り込む層。

## 5. 継続評価・改善層
選定・実行・検証結果を計測し、**モデル選定 / 能力選定 / ツール選定 / 実行フロー** を改善し続ける層。

---

# 【最重要ルール】
## 1. 先に作った層は即使う
層が完成したら、次の層の実装と運用に **即投入** する。  
保留しない。

## 2. 手動前提に戻らない
ユーザーが slash command や個別機能を覚えて操作する前提にするな。  
**オーケストラ側が裏で扱う**。

## 3. 効率化を蓄積する
新しく得た能力は、その場限りで終わらせず、**以後の標準能力** にする。

## 4. 頭脳を主軸にする
追加5層は独立実装ではなく、**既存の頭脳が判断して使う手足** として作る。

## 5. 長文化しない
説明を増やすな。  
判断・実装に必要な最小構造だけを扱え。

---

# 【各層の到達条件】
## 1. 実行制御層
- AIが機能を自動発火できる
- ユーザー手動の slash command 前提が消えている
- 実行順序が固定されている

## 2. 自動品質ゲート層
- 主要作業に自動検証が入る
- 危険変更を自動検知できる
- 品質低下のまま先へ進まない

## 3. 接続・認証・権限自動化層
- 必要機能が使える状態まで自動で整う
- 接続不足で止まりにくい
- 権限設定が標準化される

## 4. 常時監視・自動同期層
- 新機能と更新を継続取得できる
- 重要変更だけを抽出できる
- 既存判断の再評価へ接続できる

## 5. 継続評価・改善層
- 実行結果を測定できる
- 選定精度を見直せる
- 改善結果が次回以降へ反映される

---

# 【実装順序】
1. **実行制御層** を作る  
2. 完成した **実行制御層を使って** 自動品質ゲート層を作る  
3. 1と2を使って 接続・認証・権限自動化層を作る  
4. 1〜3を使って 常時監視・自動同期層を作る  
5. 1〜4を使って 継続評価・改善層を作る  

---

# 【入力】
- 既存の頭脳
- 既存の知識基盤
- 既存の実行基盤
- GitHub 上のコードと履歴
- 利用可能な公式機能 / MCP / API / OSS
- 実装中に新しく見つかる有効機能

---

# 【出力】
最低限、各層ごとに以下を明示できること。

- `goal`
- `inputs`
- `core_logic`
- `activation_rule`
- `verification`
- `handoff`

---

# 【完了条件】
以下を満たしたとき、この追加要件は成立している。

1. 5層の優先順位が固定されている
2. 既存の頭脳を前提にしている
3. 各層が完成後すぐ再利用される
4. 後続実装が前段の成果を使って進む
5. 手動依存を減らす方向に統一されている
6. 効率化と品質向上が同時に進む構造になっている

- 完成した層を寝かせるな
- 手動操作を標準にするな
- 効率化だけで品質を落とすな
