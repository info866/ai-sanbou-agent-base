# 知識層定義・構造化/文書切り分け（Knowledge Layers & Separation）

> **ステップ対応**: フェーズ3作業指示書 ステップ3（知識層振り分け）+ ステップ4（構造化/文書切り分け）  
> **最終更新**: 2026-04-13

---

## この文書の役割

知識を一枚岩で扱わず、用途ごとに層分けし、構造化で扱うべきものと文書知識で扱うべきものを明確に切り分ける。

---

## 正本との対応

要件定義書（7.フェーズ3 要件定義書.md）では K1〜K4 の表記を使用している。本文書では実装上の明瞭性のため KL-1〜KL-4 の表記を使用するが、対応は以下の通り。

- K1 = KL-1（構造化メタデータ層）
- K2 = KL-2（判断記録層）
- K3 = KL-3（文書知識層）
- K4 = KL-4（再確認要知識層）

---

## パート1｜知識層定義

### KL-1（= K1）｜構造化メタデータ層

**扱うもの**: 一意に特定可能で、プログラム的に検索・フィルタ・ソートが必要な項目。

| 対象 | 由来 | 例 |
|------|------|-----|
| 候補ID・正式名称 | phase1/02 | F-001, Prompt Improver |
| 種別コード | phase1/03 | T01〜T11 |
| 機能層コード | phase1/03 | L01〜L08 |
| 管理状態 | phase1/03 | 未確認/調査中/候補/採用候補 等 |
| 採用状態 | phase2/05 | D1〜D5 |
| 自作/流用判断 | phase2/05 | B1〜B5 |
| 優先順位 | phase2/05 | P1〜P5 |
| 評価スコア（10軸） | phase2/04 | E1=4, E2=5 等 |
| 最終確認日 | phase1/02 | 2026-04-12 |
| 再確認フラグ | 本フェーズで新設 | true/false |
| 信頼区分 | 本フェーズで新設 | T1〜T4 |
| 提供元 | phase1/02 | Anthropic, LastMile AI 等 |
| beta/GA状態 | phase1/02 | beta/public_beta/ga/experimental |
| 一次情報URL | phase1/01, 02 | URL |

**扱い方**: 構造化された項目として保持し、フィルタ・ソート・条件検索の対象とする。自然言語だけで状態を持たない。

### KL-2｜判断記録層

**扱うもの**: 採用・保留・不採用等の判断に至った理由、比較結果、根拠の記述。

| 対象 | 由来 | 例 |
|------|------|-----|
| 判断理由要約（reason_summary） | phase2/04 | 「32k starsの実績あるClaude Code特化router。LiteLLMとの使い分けは...」 |
| 比較対象（compared_with） | phase2/04 | F-022, F-036 |
| 未解決論点（unresolved_points） | phase2/04, 06 | 「Agent SDK単体でのorchestration能力の限界値」 |
| 再確認条件（next_review_condition） | phase2/04 | 「betaヘッダーが不要になりGAになったとき」 |
| 次アクション（next_action） | phase2/04 | 「LiteLLMとの比較評価を実施」 |
| 保留理由と解消条件 | phase2/05 | 各D3候補の保留理由 |
| 自作/流用の選定理由 | phase2/05 | 「pip installで全面流用可能」 |
| 比較グループ評価結果 | phase2/04 | CG-01〜CG-06の評価記述 |

**扱い方**: テキスト記述だが、構造化メタデータ層のitem_idとリンクし、候補ID指定で引けるようにする。判断記録だけを単独で全文検索する場面は少ない。候補を特定してからその判断根拠を引く流れが主。

### KL-3｜文書知識層

**扱うもの**: 長文の説明・要約・調査メモ・公式ドキュメント要点・実装パターン説明。

| 対象 | 由来 | 例 |
|------|------|-----|
| README要約 | 各リポジトリ | 「LiteLLMは100+ LLMへの統一ゲートウェイ...」 |
| 公式ドキュメント要点 | A-001〜A-015 | Hooks仕様の要約、Skills仕様の要約 |
| 比較メモ | phase2/04 | CG-01 router比較の詳細記述 |
| 実装パターン説明 | phase2/03 | Agent SDKのworkflowパターン |
| 調査メモ | phase2/03 | GitHub追加探索の結果記述 |
| フェーズ間の引き継ぎ記述 | phase1/05, phase2/06 | 「フェーズ3で知識資産として活かすべき候補」 |

**扱い方**: RAG（ベクトル検索）の主対象。チャンク分割して検索可能にする。ただしRAG結果だけで断定せず、構造化メタデータとの突合を行う。

### KL-4｜再確認要知識層

**扱うもの**: 鮮度に依存する知識で、保存時点の情報が現在も正確かどうかを保証できないもの。

| 対象 | 由来 | 例 |
|------|------|-----|
| 最新機能情報 | A-003, A-004 | Claude 4.6の新機能、Managed Agents beta仕様 |
| beta/GA状態 | phase1/02, phase2/04 | F-006: beta（2026-04-08時点） |
| pricing情報 | 各公式ドキュメント | Advisor Toolのpricing（未公表） |
| OSS活性度 | GitHub | F-023 mcp-eval: 20 stars、7ヶ月更新なし |
| release notes依存知識 | A-003, A-004 | 「2026-04-09にAdvisor Toolが公開された」 |
| 外部サービス依存仕様 | 各公式 | MCP Registry APIのエンドポイント仕様 |
| 競合・代替候補の現況 | GitHub | AutoGenのコスト改善状況 |

**扱い方**: 必ず再確認フラグ（recheck_required_flag）と再確認条件（recheck_condition）を付与する。この層の知識を根拠に重要判断を行う前は、公式一次情報またはGitHubで現在の状態を再取得する。

## パート3｜構造化対象と文書対象の切り分け

### 3-1 構造化で扱うべきもの（KL-1に格納）

以下はプログラム的なフィルタ・ソート・条件検索の対象として構造化する。

- item_id（一意識別子）
- item_name（正式名称）
- item_type（種別コード T01〜T11）
- layer_category（機能層コード L01〜L08）
- current_status（管理状態）
- decision_state（採用状態 D1〜D5）
- build_vs_buy_state（自作流用判断 B1〜B5）
- current_priority（優先順位 P1〜P5）
- evaluation_scores（E1〜E10のスコア値）
- beta_or_ga（beta/public_beta/ga/experimental）
- vendor_owner（提供元）
- source_url（一次情報URL）
- last_checked_at（最終確認日）
- recheck_required_flag（再確認要否 true/false）
- trust_level（信頼区分 T1〜T4）
- freshness_level（鮮度レベル）

### 3-2 文書知識で扱うべきもの（KL-3に格納）

以下は自然言語テキストとして保持し、RAGまたは全文検索の対象とする。

- reason_summary（判断理由の文章）
- 比較メモ（長文の比較記述）
- README要約
- 公式ドキュメントの要点説明
- 実装パターンの説明
- 調査メモ
- 参考設計思想の記述
- 引き継ぎ記述

### 3-3 混在禁止ルール

1. **状態や優先順位を自然言語だけで持たない**: 「たぶん採用」ではなくD1/D2等のコードで管理する
2. **長文比較メモを構造化項目だけで潰さない**: 比較の詳細はKL-2/KL-3に自然言語で残す
3. **構造化項目と文書知識を必ずitem_idで紐づける**: 文書知識が孤立しないようにする
4. **鮮度依存情報は構造化フラグで管理する**: recheck_required_flag, last_checked_at を構造化項目で持ち、KL-4の知識には必ず付与する
