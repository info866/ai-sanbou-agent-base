# 再確認ルール・信頼区分定義（Recheck Rules & Trust Levels）

> **ステップ対応**: フェーズ3作業指示書 ステップ8（再確認ルール）+ ステップ9（信頼区分）  
> **最終更新**: 2026-04-13

---

## この文書の役割

古くなりうる知識を安全に扱うための再確認条件を固定し、知識の強さを区別して使い方を変えられるようにする。

---

## パート1｜再確認ルール

### 1-1 鮮度依存再確認

以下の知識は時間経過により事実でなくなる可能性がある。保存時点の情報をそのまま現在の事実として使ってはならない。

| 再確認対象 | 例 | 再確認タイミング | 確認方法 |
|-----------|-----|----------------|---------|
| 最新機能情報 | Claude 4.6の機能、新API | 利用判断の前 | A-003 リリースノート確認 |
| 仕様変更 | API形式変更、パラメータ追加 | 実装着手の前 | A-001, A-002 公式ドキュメント確認 |
| beta/GA状態 | Managed Agents beta、Advisor Tool beta | 月1回 + 採用判断の前 | A-003 リリースノート確認 |
| pricing情報 | モデル料金、API料金 | コスト試算の前 | 公式料金ページ確認 |
| OSS活性度 | stars数、最終更新日、issue対応 | 採用判断の前 + 3ヶ月毎 | GitHub リポジトリ確認 |
| 競合・代替候補の現況 | 新OSS出現、既存OSSの停止 | 比較評価の前 | GitHub検索 |

### 1-2 重要判断前再確認

以下の場面では、関連する知識の鮮度にかかわらず再確認を義務とする。

| 判断場面 | 再確認対象 | 理由 |
|---------|-----------|------|
| 採用判断に進む前 | 対象候補のbeta/GA状態、一次情報URL、最新仕様 | 古い情報に基づく採用は手戻りリスクが高い |
| 実装投入判断に進む前 | 対象候補のAPI仕様、breaking change履歴、依存関係 | 実装後のbreaking changeは致命的 |
| 比較評価を行う前 | 比較対象の全候補の最新状態 | 片方だけ古い情報で比較すると結果が歪む |
| フェーズ移行時 | P1（最優先）とP2（高優先）の全候補 | フェーズ間で状況が変わっている可能性がある |

### 1-3 根拠不足再確認

以下の条件に該当する知識は、根拠が弱いため再確認対象とする。

| 条件 | 判定基準 | 対応 |
|------|---------|------|
| 比較が少ない | compared_withが空、または1件のみ | 追加の代替候補を探索してから判断する |
| 情報源が弱い | evidence_linksに一次情報がない | 一次情報で裏取りする |
| 断片情報しかない | summaryが1文以下、またはnotes頼み | 公式ドキュメント・READMEを読み直す |
| 最終確認日が古い | last_checked_atが90日以上前 | 一次情報の最新状態を取得する |

### 1-4 RAG単独断定禁止

- RAG検索結果だけで最新事実を断定しない
- RAG結果を使う場合は必ずKL-1の構造化情報で裏付ける
- 鮮度依存情報がRAGから返された場合、公式一次情報またはGitHubで再確認する

### 1-5 再確認を省略してよい条件

全知識を毎回再確認するのは非効率である。以下の条件を満たす知識は再確認を省略してよい。

| 条件 | 理由 |
|------|------|
| trust_level = T1（一次確定知識）かつ last_checked_at が30日以内 | 公式確定情報で最近確認済み |
| decision_state = D4（不採用）で再検討条件が成立していない | 不採用判断済みで状況変化なし |
| 運用ルール・分類体系の定義文書 | プロジェクト内で固定した規約であり外部鮮度に依存しない |

## パート2｜信頼区分定義

### T1｜一次確定知識

| 項目 | 内容 |
|------|------|
| 何を含むか | 公式ドキュメントから直接確認した仕様、GA済み機能の動作、プロジェクト内で固定した定義（分類軸、評価基準、状態遷移等） |
| どこまで断定してよいか | 事実として断定可能。「〇〇はGA済みの公式機能である」等 |
| 再確認の要否 | 通常不要。ただし6ヶ月以上経過した場合は念のため確認を推奨 |
| 後続フェーズでの扱い | そのまま実装の前提として使用可能 |

**該当する知識の例**:
- MCP（F-002）はオープン標準プロトコルである
- Hooks（F-004）はGA済みで24+イベント・4実装タイプに対応する
- 評価基準は10軸（E1〜E10）5段階で定義されている

---

### T2｜判断済み知識

| 項目 | 内容 |
|------|------|
| 何を含むか | フェーズ2で比較・評価を経て決定された判断（採用状態、自作/流用判断、優先順位、比較結果） |
| どこまで断定してよいか | 判断結果として参照可能。ただし「再評価条件が成立した場合は見直し対象になる」ことを併記する |
| 再確認の要否 | 判断の前提となった鮮度依存情報が変化した場合に再評価する |
| 後続フェーズでの扱い | 実装計画の入力として使用する。ただし実装着手前に前提条件の鮮度を確認する |

**該当する知識の例**:
- promptfoo（F-032）は採用候補（D1）、全面流用（B1）、高優先（P2）
- mcp-eval（F-023）は不採用（D4）。理由: 20 stars、7ヶ月更新なし
- orchestration比較（CG-04）ではLangGraphが最有力だが、Agent SDK単体との限界比較が未実施

---

### T3｜参考知識

| 項目 | 内容 |
|------|------|
| 何を含むか | README要約、実装パターンの説明、設計思想の記述、二次情報からのヒント |
| どこまで断定してよいか | 参考情報として提示可能だが、これだけを根拠にした断定は不可。一次情報での裏取りを推奨する |
| 再確認の要否 | 判断の直接根拠にする場合は一次情報で確認する |
| 後続フェーズでの扱い | 実装の参考として使用するが、実装前に一次情報を確認する |

**該当する知識の例**:
- LangGraphはグラフベースの柔軟なオーケストレーション基盤である（README要約）
- claude-agent-sdk-demosにはemail assistantとresearch agentの参照実装がある（README要約）
- executor+advisorパターンはAdvisor Toolの推奨ユースケースである（公式ドキュメント要点）

---

### T4｜再確認前提知識

| 項目 | 内容 |
|------|------|
| 何を含むか | beta/experimental機能の仕様、pricing未公表の機能、活性度が不安定なOSS、release notes依存の最新情報、外部サービスの現在の提供条件 |
| どこまで断定してよいか | 「〇〇時点では△△であった」のように時点限定で言及可能。現在の事実としての断定は不可 |
| 再確認の要否 | 利用・判断のたびに必ず一次情報で再確認する |
| 後続フェーズでの扱い | 実装着手前に必ず最新状態を確認する。確認後にT1またはT2へ昇格させる |

**該当する知識の例**:
- Managed Agents（F-006）は2026-04-08時点でpublic beta
- Advisor Tool（F-012）は2026-04-09時点でpublic beta、pricing未公表
- Agent Teams（F-008）は2026-04-12時点でexperimental
- mcp-eval（F-023）は2026-04-12時点で20 stars、7ヶ月更新なし

## パート3｜信頼区分の付与方法

### 付与の判定フロー

```
知識を取得する
  │
  ├─ 公式一次情報から直接確認 → GA済み → T1
  │                            → beta/experimental → T4
  │
  ├─ フェーズ2の比較・評価を経た判断 → T2
  │
  ├─ README要約・実装パターン・調査メモ → T3
  │
  └─ 鮮度依存・外部サービス依存 → T4
```
