# 知識活用フロー定義（Knowledge Utilization Flow）

> **ステップ対応**: フェーズ3作業指示書 ステップ6  
> **最終更新**: 2026-04-13

---

## この文書の役割

問いが来たときに、どの順で知識を見て、いつ外部再確認するかを固定する。後続フェーズのエージェントや人間が、知識基盤をどう使えばよいかに迷わないようにする。

---

## 知識活用フローの全体像

```
問い受付
  │
  ├─ F1: 問い合わせの分類
  │    └─ SR-01〜SR-07のどれに該当するか
  │
  ├─ F2: 知識源の選択
  │    └─ 構造化メタデータ / 判断記録 / 文書知識 / 外部再調査 のどれを優先するか
  │
  ├─ F3: 検索実行
  │    └─ KL-1 → KL-2 → KL-3 の順で必要な知識を取得
  │
  ├─ F4: 根拠紐付け
  │    └─ 検索結果に元情報源・元比較・元判断を紐付ける
  │
  ├─ F5: 鮮度判定
  │    └─ recheck_required_flag, trust_level を確認し、必要時は外部再確認
  │
  └─ F6: 出力形成
       └─ 確定 / 判断済み / 再確認必要 / 未解決 に分けて返す
```

---

## 各ステップの詳細

### F1｜問い合わせの分類

問いを受け取ったら、以下のどれに該当するかを判定する。

| 問いの種類 | 検索要件ID | 判定条件 |
|-----------|-----------|---------|
| 特定候補の詳細 | SR-01 | 候補名またはitem_idが含まれている |
| 作業に効く候補の探索 | SR-02 | 作業目的や機能層の指定がある |
| 判断理由の確認 | SR-03 | 「なぜ」「理由」等の判断根拠を問う |
| 比較結果の確認 | SR-04 | 「比較」「どちらが」等の比較を問う |
| 再確認対象の抽出 | SR-05 | 「再確認」「最新」「古い」等の鮮度を問う |
| 未解決論点の確認 | SR-06 | 「未解決」「保留」「まだ」等の状態を問う |
| 公式根拠の裏取り | SR-07 | 「公式」「仕様」「正本」等の一次情報を求める |
| 複合問い | 複数SR | 上記の組み合わせ → 主軸のSRから順に処理 |

---

### F2｜知識源の選択

問い合わせ内容に応じて、どの知識層を優先的に参照するかを決定する。

| 検索要件 | 優先知識源 |
|---------|-----------|
| SR-01（既知候補） | KL-1 → KL-2 → KL-3 |
| SR-02（用途別） | KL-1でフィルタ → KL-2で補足 |
| SR-03（判断根拠） | KL-2を直接参照 |
| SR-04（比較結果） | KL-2 → KL-1でスコア比較 |
| SR-05（再確認対象） | KL-4 → KL-1 |
| SR-06（未解決論点） | KL-2 → KL-4 |
| SR-07（公式根拠） | KL-1のsource_url → KL-3の公式要点 |

---

### F3｜検索実行

F2で決定した知識源に対して検索を実行し、関連度の高いものを抽出する。

**KL-1（構造化メタデータ）の検索**: item_id、item_type、layer_category、decision_state等でフィルタ・ソート

**KL-2（判断記録）の検索**: 候補を特定した上で、reason_summary、compared_with、unresolved_pointsを取得

**KL-3（文書知識）の検索**: RAGまたは全文検索で、README要約、比較メモ、公式要点等を取得

---

### F4｜根拠紐付け

検索結果に対して、元情報源・元比較・元判断を紐付ける。

**紐付けるべき根拠**:
- evidence_links: 判断の根拠となった公式URL・GitHubリポジトリ
- originating_comparisons: どの比較グループ（CG-01等）から生まれた知識か
- originating_decisions: どの判断（D1採用候補、D4不採用等）に基づくか
- source_item_id: 元となる候補のitem_id

**根拠紐付けの義務**:
- 根拠のない検索結果を高信頼として返さない
- evidence_linksが空の知識は、trust_levelをT3以下で扱う
- 公式根拠検索（SR-07）では、evidence_linksの中から公式一次情報を優先的に返す

---

### F5｜鮮度判定

取得した知識の鮮度を判定し、必要時は外部再確認する。

**鮮度判定ルール**:
1. recheck_required_flag を確認する
2. last_checked_at が90日以上前なら再確認推奨
3. trust_level が T4なら必ず鮮度注記を付ける
4. beta/experimentalの候補は常に鮮度注記を付ける

**鮮度判定の結果分類**:

| 判定 | 条件 | 対応 |
|------|------|------|
| 鮮度問題なし | recheck_required_flag = false かつ last_checked_at が90日以内 | そのまま回答 |
| 再確認推奨 | recheck_required_flag = true、またはlast_checked_atが90日超過 | 回答に「再確認推奨」注記を付記 |
| 要外部確認 | 重要判断の前提になる鮮度依存情報 | 公式一次情報 or GitHub を再取得 |

**外部再確認後の義務**:
- KL-1のlast_checked_atを更新する
- 変化があった場合はKL-2に変更理由を記録する
- recheck_required_flagを適切に更新する

---

### F6｜出力形成

知識を要約し、確信度ラベルを付けて返す。

| ラベル | 意味 | 条件 |
|--------|------|------|
| **確定** | 構造化メタデータに基づく事実。変更の余地が小さい | KL-1のGA済み公式機能の仕様、確定済みの判断結果 |
| **判断済み** | フェーズ2で比較・評価を経て決めた判断。根拠あり | KL-2の判断記録に基づく回答 |
| **再確認必要** | 鮮度依存情報に基づいており、最新状態の確認を推奨 | KL-4所属、または鮮度判定で再確認推奨となった情報 |
| **未解決** | まだ決着していない論点。回答は暫定的 | KL-2のunresolved_points、decision_state = D3 |

**出力フォーマット例**:
```
[判断済み] F-032 promptfooは採用候補（D1）。
理由: 20k stars、Anthropic公式使用、YAML宣言的設定でモデル比較が容易。
比較対象: deepeval（F-033）、mcp-eval（F-023）
[再確認必要] Node.js v22.22+への更新が前提条件として残っている。
```

---

## フロー適用の絶対ルール

1. **F1〜F2は必ず実行する**: 問いの分類と知識源の選択を飛ばさない
2. **問いごとに場当たりで知識源を選ばない**: 上記フローに従う
3. **F4（根拠紐付け）を省略しない**: 根拠のない知識を高信頼で返さない
4. **古い知識をそのまま返さない**: F5の鮮度判定を必ず通す
5. **確信度ラベルを省略しない**: F6の出力形成で必ず付与する
6. **KL-3（文書知識）だけで回答しない**: 必ずKL-1と突合する
7. **RAG結果だけで断定しない**: RAGはKL-3の検索補助であり、KL-1の構造化情報で裏付ける
