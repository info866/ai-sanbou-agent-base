# RAG適用範囲定義（RAG Scope）

> **ステップ対応**: フェーズ3作業指示書 ステップ7  
> **最終更新**: 2026-04-13

---

## この文書の役割

RAGをどこに使い、どこに使わないかを固定する。RAGは知識活用の万能解決策ではなく、文書知識層（KL-3）の検索補助として位置づける。

---

## RAGの位置づけ

RAGは本プロジェクトにおいて以下の通り位置づける。

- **RAGの役割**: 文書知識層（KL-3）に格納された長文テキスト（README要約、比較メモ、公式ドキュメント要点、調査ノート等）を意味検索可能にする補助手段
- **RAGでないもの**: 状態管理や判断確定情報の正本。構造化メタデータ（KL-1）の代替。唯一の検索手段
- **RAGと併用すべきもの**: 構造化メタデータ検索（KL-1のフィルタ・ソート）、判断記録の直接参照（KL-2）

## パート2｜RAG非向き / 構造化優先資産

以下はRAGではなく構造化メタデータ（KL-1）として管理し、フィルタ・ソート・条件検索で扱う。

| 項目 | 理由 |
|------|------|
| item_id（一意識別子） | 完全一致検索が必要。ベクトル検索は不適切 |
| decision_state（D1〜D5） | 離散値。フィルタ検索が最適 |
| build_vs_buy_state（B1〜B5） | 離散値。フィルタ検索が最適 |
| current_priority（P1〜P5） | 離散値。ソートが最適 |
| evaluation_scores（E1〜E10のスコア） | 数値。条件検索・ソートが最適 |
| beta_or_ga（状態コード） | 離散値。フィルタ検索が最適 |
| recheck_required_flag（true/false） | ブーリアン。条件検索が最適 |
| trust_level（T1〜T4） | 離散値。フィルタ検索が最適 |
| last_checked_at（日付） | 日付。範囲検索が最適 |
| source_url（URL） | 完全一致参照が必要 |

## パート3｜RAG利用ルール

### ルール1: RAGだけで断定しない

RAG検索結果を唯一の根拠として重要判断を行わない。必ず以下と突合する。
- KL-1の構造化メタデータで候補の現在状態を確認する
- 鮮度依存情報の場合、recheck_required_flagを確認する

### ルール2: 鮮度依存情報は再確認前提

RAGから取得した情報がKL-4（再確認要知識層）に属する場合、その情報の取得時点を確認し、必要に応じて公式一次情報で再取得する。

### ルール3: 出典追跡可能な状態で使う

RAGから取得した情報には、必ず以下を付記する。
- 元の知識ID（KI-xxx）
- 元の格納先ファイルパス
- 元の情報源（source_id / source_url）

### ルール4: 構造化情報と併用する

RAGの検索結果を返す際は、該当候補のKL-1メタデータ（採用状態、優先順位、信頼区分）を併記する。文書知識だけを返さない。

### ルール5: RAGインデックスの更新

知識資産が更新された場合（新候補追加、再評価、状態変更等）、対応するRAGインデックスも更新する。古いインデックスで検索した結果が現在の状態と矛盾する場合は、構造化メタデータを優先する。

## パート4｜RAG導入判断

### 現フェーズでの判断

フェーズ3は知識活用基盤の**設計**フェーズであり、本番RAGの実装は行わない。

RAG導入のタイミング判断:
- **フェーズ4（実行基盤フェーズ）**: エージェントが知識基盤を参照する仕組みの一部としてRAGの実装を検討する
- **導入判断の前提条件**: 文書知識層（KL-3）の資産量が十分（目安: チャンク数100以上）になった時点で実装開始する

### RAG技術選定の方向性（フェーズ4以降で確定）

本フェーズでは技術選定を確定しないが、以下の方向性を記録する。
- ベクトルDB: フェーズ4でLangfuse/Arize Phoenix等のオブザーバビリティ基盤選定と合わせて検討
- エンベディングモデル: Anthropic公式のembedding API、またはオープンソースモデル
- チャンク戦略: 候補単位・セクション単位のハイブリッド
