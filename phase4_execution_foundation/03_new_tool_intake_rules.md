# 新規ツール即時投入ルール（New Tool Immediate Intake Rules）

> **ステップ対応**: フェーズ4作業指示書 ブロック3  
> **最終更新**: 2026-04-13

---

## この文書の役割

フェーズ4作業中に新しく見つかったツール・MCP・API・OSS を、いかに素早く評価し、使えるなら即座に作業フローへ組み込むかのルール。「あとで検討」を前提にしない。

---

## 原則1：「あとで検討」は標準にしない

### 禁止するパターン

以下は**禁止**する。代わりに「その場で判定」を実施：

❌ 「いいツールを見つけたけど、忙しいから後で検討しよう」
❌ 「新規候補があるが、現在の計画が終わるまで待とう」
❌ 「効果がありそうだが、リスクを考えてPOC段階で留め置こう」

### 許可するパターン

以下だけは許可する（明確な理由がある場合）：

✅ 「このツールは今回の作業には関係ないが、次フェーズで評価する」（判定は「フェーズ[X]で再評価」と明記）
✅ 「このツール自体は高価値だが、所要時間が大きく現在の重要度と合わない。ただし小規模試験なら即実施」（実行計画を立てる）
✅ 「この候補は外部API・サービスが不安定なため、本格導入は見送るが、仕様確認だけは今週末に実施」（着手時期を明記）

## ルール1：新規ツール発見時の判定フロー（Intake Decision Flow）

### フロー全体

```
新規ツール発見
  ↓
【判定1】今回の作業に直接効くか？
  ├─ YES → 【判定2へ】
  └─ NO  → 「フェーズ[X]で再評価」と記録 → [END]

【判定2】既存フローより改善できるか？
  ├─ YES → 【判定3へ】
  └─ NO  → 「優位性なし」と記録 → [END]

【判定3】すぐ使えるか？（セットアップ時間 < 30分）
  ├─ YES → 「即投入」判定 → [段階2へ]
  └─ NO  → 【判定4へ】

【判定4】セットアップ時間が許容か？（30分〜2時間）
  ├─ YES → 「試験導入」判定 → [段階2へ]
  └─ NO  → 「学習フェーズ必須」と記録 → [フェーズ[X]へ]

### 段階2：投入方法の決定
即投入 or 試験導入に決定 → 03_new_tool_intake_workflow を実施
```

### 判定フロー詳細

#### 【判定1】今回の作業に直接効くか？

**質問**: このツールは、現在進行中のタスク・ステップで具体的に役に立つか？

**評価基準**:
- **YES**: 現在のステップ（調査/比較/実装/修正/検証/記録/GitHub）に直接利用可能
- **NO**: 関連性あるが間接的、または別タスク向け

**記録フォーマット** (NO の場合):
```
新規ツール: [ツール名]
判定1結果: NO
理由: [具体的な理由]
推定適用フェーズ: フェーズ[X]（[いつ頃の作業で必要か]）
→ 記録: phase4_execution_foundation/new_tool_tracking.md に記載
```

#### 【判定2】既存フローより改善できるか？

**質問**: 現在使用予定の D1・D2・P1・P2 候補より、このツールの方が効果的か？

**評価基準**:
- **YES**: 実装時間短縮 / 品質向上 / 自動化度向上 / コスト削減 のいずれかが明確
- **NO**: 既存候補と同等以下の効果

**定量的な比較例**:
```
現在の計画: promptfoo（セットアップ15分、評価自動化）
新規候補: [ツール名]
比較結果:
  - セットアップ時間: [時間] vs 15分 → [改善度]
  - 評価メトリクス: [数値] vs promptfooの[数値] → [改善度]
  - 学習コスト: [時間] vs 5分 → [改善度]
判定2結果: [YES/NO]
```

**記録フォーマット** (NO の場合):
```
新規ツール: [ツール名]
判定2結果: NO（既存候補 [F-XXX] と同等以下）
理由: [具体的な理由]
→ 記録に留める。本格導入は保留
```

#### 【判定3】すぐ使えるか？（セットアップ時間 < 30分）

**質問**: インストール＆初期設定＆簡単な動作確認が 30 分以内で終わるか？

**評価基準**:
- **YES**: `pip install [ツール]` or `npm install` or API key設定だけで使用開始可能
- **NO**: ドキュメント読み込み・複雑な設定・依存関係解決が必要

**セットアップ時間の目安**:

| パターン | 時間 | 例 |
|---------|------|-----|
| **即座に使用可能** | 1〜5分 | `pip install`, Console API, `/commandだけで使用可能 |
| **簡単セットアップ** | 5〜15分 | API key設定、YAMLで基本設定、venv設定 |
| **標準セットアップ** | 15〜30分 | Docker setup、複数ファイル設定、依存関係調整 |
| **複雑セットアップ** | 30分超 | DB初期化、複雑なコンフィグ、ドキュメント精読必須 |

**判定3で YES の場合**: 判定3を完了したら即座に段階2へ（試験導入を開始）

**記録フォーマット** (NO の場合):
```
新規ツール: [ツール名]
判定3結果: NO（セットアップ時間: [時間]分）
セットアップ内容: [何に時間がかかるのか]
→ 判定4へ
```

#### 【判定4】セットアップ時間が許容か？（30分〜2時間）

**質問**: 現在の優先度を考慮して、セットアップに 30分〜2時間費やす価値があるか？

**評価基準**:
- **YES**: セットアップ時間に見合う効果が期待でき、現在の重要度に合致する
- **NO**: セットアップコストが大きく、現在は後回しすべき

**優先度マトリクス**:

| 効果 × セットアップ時間 | 30〜60分 | 60〜120分 | 120分超 |
|----------------------|---------|---------|--------|
| **高効果（大幅改善）** | YES ✅ | YES ✅ | 要検討 |
| **中効果（部分改善）** | YES ✅ | 要検討 | NO ❌ |
| **低効果（参考程度）** | NO ❌ | NO ❌ | NO ❌ |

**記録フォーマット** (YES の場合):
```
新規ツール: [ツール名]
判定4結果: YES
セットアップ時間: [時間]分
期待効果: [セットアップコストに見合う効果内容]
計画: [今週中に試験導入] or [来週から試験導入]
→ 段階2へ
```

**記録フォーマット** (NO の場合):
```
新規ツール: [ツール名]
判定4結果: NO（セットアップコストが大きすぎる）
推定適用フェーズ: フェーズ[X]
理由: [具体的な理由]
→ 記録に留める。本格導入は後回し
```

## ルール2：判定時間の制限（タイムボックス）

### タイムボックス原則

新規ツール発見から最終判定までの時間を制限する。「あとで検討」を避けるため：

**判定1〜4をすべて完了**: **15分以内**

- 各判定に 3分以内を費やす
- 不確実な情報は「フェーズXで再評価」と判定
- 「もっと詳しく知りたい」は試験導入フェーズで実施

### タイムボックス超過時の対応

15分を超えても判定できない場合：
1. 現時点での判定結果（YES/NO/保留）を記録
2. 「詳細確認は試験導入フェーズで実施」と記載
3. 試験導入の計画（いつ、誰が、何を確認するか）を立てる

---

## ルール3：試験導入フローと判定基準

### 即投入 vs 試験導入の区分

**即投入** (判定3 = YES, セットアップ < 30分):
- その場で install / setup を実施
- 現在進行中のタスクに組み込む
- 1時間以内に効果実感できることを確認

**試験導入** (判定4 = YES, セットアップ 30分〜2時間):
- 小さなタスク・ワークフローで試験
- 効果・課題を記録
- 本格導入か中止かを判定

### 試験導入の標準フロー

```
【実施】
1. セットアップを完了（合意のセットアップ時間内）
2. 小規模タスク（1つのファイル、1つのワークフロー）で試験
3. 結果を記録：効果あり/課題あり/効果なし

【判定】
効果あり → 本格導入（計画に追加）
課題あり → 改善検討 → 本格導入か中止か判定
効果なし → 中止（理由を記録）
```

### 試験導入記録フォーマット

```
新規ツール試験: [ツール名]
試験対象タスク: [具体的なタスク内容]
試験期間: [実施期間]
セットアップ実績: [実際にかかった時間]
試験結果:
  - [実施内容]: [結果]
  - [効果測定]: [定量結果]
  - [課題]: [見つかった課題]
本格導入判定:
  - [ ] 本格導入へ（理由: [効果が十分]）
  - [ ] 改善後に再試験（理由: [課題内容]）
  - [ ] 中止（理由: [効果不足]）
```

## ルール4：新規ツール投入のGoal

各段階での完了条件：

### 段階1：判定完了
- [ ] 【判定1】今回の作業に直接効くか？ → YES/NO/再評価時期 を記録
- [ ] 【判定2】既存フローより改善できるか？ → YES/NO の判定を記録
- [ ] 【判定3】すぐ使えるか？ → YES/NO を記録
- [ ] 【判定4】（必要な場合）セットアップ時間が許容か？ → YES/NO を記録

### 段階2a：即投入の完了
- [ ] セットアップ完了
- [ ] 現在のタスク・ステップに組み込み
- [ ] 1時間以内の効果実感を確認
- [ ] 効果・課題・次アクションを記録

### 段階2b：試験導入の完了
- [ ] セットアップ完了
- [ ] 試験タスクの実施完了
- [ ] 本格導入 / 改善再試験 / 中止 の判定完了
- [ ] 試験結果・判定理由を記録

## ルール5：新規ツール投入の条件付き除外

以下の場合は、新規ツール投入ルールを適用せず、フェーズ4では使用しない：

### 除外条件1：セキュリティ・信頼性リスクが明らかな場合

**例**:
- 認証情報を平文で保存する要件
- 未認可の外部API呼び出しが必須
- OSS活性度が著しく低い（最終更新が1年超）

**対応**: その場で「セキュリティリスクのため不採用」と判定、理由を記録

### 除外条件2：Anthropic エコシステムと重大な競合

**例**:
- OpenAI 専用ツール（LiteLLM で統一する方針に矛盾）
- Claude Code / Claude AI と機能が重複し、標準機能の方が優先

**対応**: 「エコシステム方針に不適合」と判定

### 除外条件3：フェーズ4スコープ外のツール

**例**:
- 「フェーズ5でのみ有用」なツール（AI参謀統合フレームワーク等）
- 「フェーズ3で評価済みの保留候補」の再浮上

**対応**: 「スコープ外。フェーズXで評価」と判定

## 実装例：新規ツール発見から投入まで

### シナリオ1：モデルルーター新候補発見（即投入パターン）

```
【発見】GitHub の trending で新しい claude-code-router 派生ツールを発見

【判定フロー】
判定1: 今回の作業に直接効くか？
  → YES（複数モデル比較タスクで即役立つ）

判定2: 既存フローより改善できるか？
  → YES（LiteLLM との統合時間が短い）

判定3: すぐ使えるか？（セットアップ < 30分）
  → YES（pip install one-liner）

→ 即投入判定

【実施】
時刻: 14:45
実施内容: pip install [ツール]
  → テスト呼び出し実施（5分）
  → Claude 4.6 / 4 の出力品質比較（10分）
結果: 効果確認
  → LiteLLM 単独より [XX]%高速化確認
記録: 本格導入（現在のLiteLLM パイプラインに統合）
```

### シナリオ2：複雑な評価フレームワーク候補（試験導入パターン）

```
【発見】deepeval 新バージョンが Python-native メトリクスで話題

【判定フロー】
判定1: 今回の作業に直接効くか？
  → YES（LLM評価パイプラインで候補）

判定2: 既存フローより改善できるか？
  → YES（promptfoo は YAML 中心だが、deepeval は Python ネイティブ）

判定3: すぐ使えるか？（セットアップ < 30分）
  → NO（ドキュメント精読・複雑設定必要、推定 45分）

判定4: セットアップ時間が許容か？（30〜120分）
  → YES（prompt foo 導入後のメトリクス拡張価値あり）

→ 試験導入判定

【試験計画】
試験時期: 今週末（土）
試験対象: prompt foo の評価結果に deepeval メトリクスを追加
期待効果: Python-native メトリクスで精度向上
セットアップ時間予算: 60分

【試験実施・判定】（土に実施）
セットアップ実績: 50分
試験結果: メトリクス正常動作、ただしpromptfoo 統合に複雑性
判定: 改善後に再試験
  理由: 複数メトリクスの同時使用でワークフロー複雑化の懸念
次アクション: 今月末に「deepeval vs promptfoo metrics」の比較調査実施
```

### シナリオ3：スコープ外候補（後回しパターン）

```
【発見】LangGraph 新バージョン（experimental フラグ外れた、GA化報告）

【判定フロー】
判定1: 今回の作業に直接効くか？
  → NO（フェーズ4 執行基盤はAgent SDK 中心。LangGraph はフェーズ5向け）

判定1 NO → フェーズ 5 で再評価と記録

【記録】
新規ツール: LangGraph
発見時期: 2026-04-13
判定1結果: NO（フェーズ4 スコープ外）
推定適用: フェーズ5（AI参謀最終統合時）
記録先: phase4_execution_foundation/new_tool_tracking.md
→ 次回フェーズ移行時に再評価対象として引き継ぎ
```
