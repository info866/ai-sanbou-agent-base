# フェーズ4実行チェックリスト（Phase 4 Execution Checklist）

> **役割**: 7ステップ実行フロー（調査→比較→実装→修正→検証→記録→GitHub反映）を実装チームが確実に実行するためのガイド  
> **更新**: 2026-04-13

## タスク開始時のチェックリスト

実装タスク開始前に以下を確認：

- [ ] **対象明確**: 何をやるのかが明記されているか（1文で説明可能か）
- [ ] **スコープ確定**: In/Out の範囲が決まっているか  
- [ ] **完了条件定義**: 何ができたら終わりかが定量的に記述されているか
- [ ] **ツール候補確認**: Phase 2・3 で採用されたツール・MCP・API を確認したか
- [ ] **フェーズ4実行フロー確認**: 本チェックリストを読了したか

## STEP 1: 調査（Investigation Phase）

実行チェック項目：

### 1-1 既知知識参照
- [ ] phase3_knowledge_foundation から知識を引く
  - [ ] KL-1: 構造化メタデータ（candidate ID・採用状態・優先度）確認
  - [ ] KL-2: 判断記録（採用理由・比較結果）確認
  - [ ] KL-3: 文書知識（README・実装パターン）確認
  - [ ] KL-4: 再確認要知識（今回確認が必要な項目）確認

- [ ] trust_level（T1〜T4）を確認
  - T1: 一次確定知識 → そのまま使用可能
  - T2: 判断済み知識 → 前提条件確認が必要
  - T3: 参考知識 → 一次情報での裏取り必須
  - T4: 再確認前提 → 必ず一次情報で再確認

- [ ] last_verified_at が 90日以内か確認（古い場合は再確認ルール適用）

### 1-2 公式情報確認
- [ ] Beta/GA 状態の最新確認（リリースノート）
- [ ] API仕様の変更有無確認（公式ドキュメント）

- [ ] 競合・代替候補の現況確認（新規出現あるか）

**使用ツール**:
- [ ] WebFetch: 公式ドキュメント・リリースノート取得
- [ ] WebSearch: 最新情報・代替候補探索
- [ ] Grep/Glob: ローカル参考実装・参考コード検索

### 1-3 ツール選定

- [ ] **P1（最優先）候補**: 真っ先に検討したか

- [ ] **P2（高優先）候補**: P1で不足ならば追加検討したか
- [ ] **D2（試験導入）候補**: 該当するものを試験検討したか
- [ ] **D3・D4・P3以下**: 「今は不採用」と明示的に判定したか

**記録フォーマット** (今回の作業に使用予定のツール):

## STEP 3: 実装（Implementation Phase）

### 3-1 対象固定
- [ ] 実装するファイル・機能が明確か
- [ ] 依存関係を全て洗い出したか（import, 呼び出し, データ形式）
- [ ] 影響範囲を把握したか（他ファイルへの波及）
- [ ] **完了条件を定量的に記述したか** (重要!)
  - 例: 「X がインポートエラーなく動作」「API呼び出しが 5秒以内に完了」

### 3-2 ツール固定
- [ ] build_vs_buy_state を確認したか
  - B1: pip install で全部入り
  - B2: OSS中核 + カスタマイズ
  - B3: 複数OSS部分採用
  - B4: 参考実装参照 + 自作主体
  - B5: 0から自作

- [ ] 主ツール・副ツールを選定したか
- [ ] 自作/流用の理由をフェーズ2から確認したか

### 3-3 実装実行
- [ ] pip install / git clone / ファイルコピー / API設定 / 自作実装のいずれかを実行
- [ ] 必要なファイルを全て作成/更新したか
- [ ] requirements.txt / .gitmodules / 認証情報隔離 などを実施したか

### 3-4 自己検証（4階層）

#### 階層1: 構文検証（5分）
- [ ] Python: `python -m py_compile [file.py]` → PASS
- [ ] JavaScript: `node -c [file.js]` OR `eslint [file]` → PASS
- [ ] YAML: `yamllint [file.yaml]` → PASS
- [ ] JSON: `jq . < [file.json]` → PASS

#### 階層2: インポート・依存関係検証（5分）
- [ ] Python: `python -c "import [module]; print([module].__version__)"` → PASS
- [ ] Node: `npm ls` → PASS
- [ ] Go: `go mod verify` → PASS
- [ ] 外部API: `curl` テストコール → PASS

#### 階層3: 機能検証（10〜30分）
- [ ] Unit test が成功するか → PASS
- [ ] Manual test で想定動作するか → PASS
- [ ] エラーハンドリングが機能するか → PASS
- [ ] Edge case に対応しているか → PASS

#### 階層4: パフォーマンス・セキュリティ検証（5〜15分）
- [ ] 処理時間が許容値以内か（API < 5秒, ファイル処理 < 1秒/1000行） → PASS
- [ ] 認証情報が平文に含まれていないか（`.env` に隔離） → PASS
- [ ] SQL injection / XSS に対応しているか → PASS
- [ ] 依存ライブラリの脆弱性がないか（`pip audit`, `npm audit`） → PASS

**自己検証結果記録**:
```
自己検証:
  [ ] 階層1（構文）: PASS / FAIL
  [ ] 階層2（インポート）: PASS / FAIL
  [ ] 階層3（機能）: PASS / FAIL
  [ ] 階層4（パフォーマンス）: PASS / FAIL

FAIL が1つでもある場合:
  → STEP 4（修正）へ進む
```

## STEP 5: 検証（Verification Phase）

### 5-1 完了条件確認
STEP 3-1 で定義した完了条件をすべてチェック：

- [ ] 完了条件1: [条件] → PASS / FAIL
- [ ] 完了条件2: [条件] → PASS / FAIL
- [ ] 完了条件3: [条件] → PASS / FAIL
- [ ] ... すべて PASS か

### 5-2 影響範囲確認
- [ ] STEP 3-1 で確認した「影響範囲」に実装が悪影響を与えていないか確認
- [ ] 関連ファイルを全て特定したか
- [ ] 各関連ファイルで影響テストを実施したか
- [ ] 副作用がないことを確認したか

**副作用確認テンプレート**:
```
関連ファイル: [ファイル一覧]
各ファイルへの影響:
  - [ファイルA]: [影響内容] → [検証結果: PASS/FAIL]
  - [ファイルB]: [影響内容] → [検証結果: PASS/FAIL]
副作用有無: [あり/なし]
```

### 5-3 差分確認
- [ ] `git status` で全変更ファイル確認
- [ ] `git diff` で各ファイルの内容確認
- [ ] 意図しない変更がないか確認
- [ ] 必要な変更がすべて含まれているか確認

## STEP 7: GitHub反映（GitHub Integration）

### 7-1 コミット作成
- [ ] 変更ファイルを `git add` でステージ
- [ ] `git commit` でコミット（メッセージフォーマット参照）

**コミットメッセージフォーマット**:
```
[type]([scope]): [subject]

[body]

[footer]
```

**type** の選択:
- [ ] feat: 新機能
- [ ] fix: バグ修正
- [ ] docs: ドキュメント
- [ ] refactor: リファクタリング
- [ ] perf: パフォーマンス改善
- [ ] test: テスト追加
- [ ] chore: ビルド・ツール変更

**body** の記載:
- [ ] **What**: 何を変えたか
- [ ] **Why**: なぜ変えたか
- [ ] **How**: どう変えたか
- [ ] **参考情報**: phase2 の判断根拠・参考実装など

**footer** の記載:
- [ ] `Relates-to: phase2/CG-XX` (比較グループ)
- [ ] `Relates-to: phase3/...` (知識ベース参照)

### 7-2 PR作成（必要時）
- [ ] 複数ファイルの連携が複雑な場合か
- [ ] 重要な判断が含まれているか
- [ ] チームレビューが必要か

→ YES の場合のみ PR 作成

### 7-3 origin/main へ push
- [ ] コミット前に `git log` で履歴確認
- [ ] `git push origin main` (PR の場合は PR マージ後)
- [ ] GitHub web で反映確認

## タスク完了時の最終チェック

実装作業全体の完了を判定する前に、以下を確認：

- [ ] **全ステップ実行**: STEP 1〜7 すべて実行済み
- [ ] **自己検証完全**: 4階層検証すべて PASS
- [ ] **差分確認済み**: git diff で意図通りの変更か確認
- [ ] **完了条件満たし**: すべての完了条件が PASS
- [ ] **影響範囲確認**: 副作用がないことを確認
- [ ] **記録完全**: 使用ツール・変更内容・検証方法・次アクション全て記録
- [ ] **GitHub反映**: コミット・push 完了、origin/main に反映
- [ ] **エラー・問題隠蔽なし**: 見つかった問題は全て対応済み

✅ すべてチェック後に「完了」扱い
❌ 1つでも未チェック → STEP 3〜5 に戻る

## 実装チーム向けガイド

- **1タスク = 1ワークフロー**: 各タスク = STEP 1〜7 の完全実行
- **本チェックリストで進捗管理**: 各ステップの完了チェックボックスを mark する
- **記録は重要**: STEP 6 の記録なしで次タスクに進まない
- **GitHub は真実**: コミット・push 前に必ず git diff で確認
- **質問・疑問は Phase 4 ドキュメント参照**: 不明な点は 01〜07 ファイルを参照
