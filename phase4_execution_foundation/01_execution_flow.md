# 実行フロー定義（Execution Flow Definition）

> **ステップ対応**: フェーズ4作業指示書 ブロック1  
> **最終更新**: 2026-04-13

---

## この文書の役割

調査から GitHub反映までの完全フローを定義し、各ステップでの判断・検証・知識参照を明記する。

---

## 全体フロー（7ステップ）

```
[START] 
  ↓
[STEP 1: 調査]    (Investigation Phase)
  ├─ 既知知識参照
  ├─ 公式情報確認
  └─ ツール選定
  ↓
[STEP 2: 比較]    (Comparison Phase)
  ├─ 比較対象選定
  ├─ 評価基準適用
  └─ 採用判断
  ↓
[STEP 3: 実装]    (Implementation Phase)
  ├─ 対象固定
  ├─ ツール固定
  ├─ 実装実行
  └─ 自己検証
  ↓
[STEP 4: 修正]    (Fix Phase)
  ├─ 問題把握
  ├─ 修正実行
  └─ 再検証
  ↓
[STEP 5: 検証]    (Verification Phase)
  ├─ 完了条件確認
  ├─ 影響範囲確認
  └─ 差分確認
  ↓
[STEP 6: 記録]    (Recording Phase)
  ├─ 何を使ったか記録
  ├─ 何を変えたか記録
  ├─ どう検証したか記録
  ├─ 未完了の記録
  └─ 次アクション記録
  ↓
[STEP 7: GitHub反映] (GitHub Integration)
  ├─ コミット作成
  ├─ PR作成（必要時）
  └─ origin/main へ push
  ↓
[END]
```

---

## ステップ詳細

### STEP 1: 調査（Investigation Phase）

**目的**: 何をやるべきかを判断するための情報収集

#### 1-1 既知知識参照

**実行内容**:
- phase3_knowledge_foundation/ から以下を引く：
  - KL-1: 構造化メタデータ（候補ID・採用状態・優先順位・信頼区分）
  - KL-2: 判断記録（採用理由・比較結果・未解決論点）
  - KL-3: 文書知識（README要約・実装パターン）
  - KL-4: 再確認要知識（今回確認が必要な項目）

**参照対象**:
- phase2_decision_foundation/05_adoption_build_vs_buy_priority.md（採用状態）
- phase2_decision_foundation/04_candidate_evaluations.md（評価スコア）
- phase3_knowledge_foundation/06_recheck_rules_and_trust.md（信頼区分）

**判定ポイント**:
- 対象候補の trust_level（T1〜T4）を確認
  - T1（一次確定知識）: そのまま使用可能
  - T2（判断済み知識）: 実装前に前提条件確認
  - T3（参考知識）: 一次情報での裏取り必須
  - T4（再確認前提知識）: 必ず一次情報で再確認
- last_verified_at が古い（90日以上）か確認
- recheck_required_flag = true か確認

**チェックリスト**:
- [ ] 候補の採用状態（D1〜D5）確認
- [ ] 優先順位（P1〜P5）確認
- [ ] 信頼区分（T1〜T4）確認
- [ ] last_verified_at 確認
- [ ] 比較対象候補確認

#### 1-2 公式情報確認

**実行内容**:
- phase3_knowledge_foundation/06_recheck_rules_and_trust.md の再確認ルール（C1〜C4）に基づき、必要な公式確認を実施

**確認対象（優先順位順）**:

| 再確認ルール | 確認タイミング | 確認方法 |
|-----------|---------------|---------|
| **C1: 鮮度依存再確認** | 利用判断の前 | リリースノート・公式ドキュメント |
| **C3: 重要判断前再確認** | 採用判断前 | 一次情報URL確認 |
| **C2: 根拠不足再確認** | 比較が少ない場合 | 追加代替候補探索 |
| **C4: RAG単独断定禁止** | RAG結果を使う場合 | KL-1構造化情報で裏付け |

**具体的確認リスト**:
- [ ] Beta/GA状態の最新確認
- [ ] API仕様の変更有無
- [ ] Pricing情報の最新確認
- [ ] OSS活性度（stars, 更新日）確認
- [ ] 競合・代替候補の現況確認

**Tool選定** （以下から選択）:
- **WebFetch**: 公式ドキュメント・リリースノート取得
- **WebSearch**: 最新情報・代替候補探索
- **Grep/Glob**: ローカル参考実装・参考コード検索
- **GitHub**: リポジトリ確認（stars, issues, 最終更新）

#### 1-3 ツール選定

**実行内容**:
- phase2_decision_foundation/ の採用候補から、今回の作業に効くツール・MCP・APIを選定

**選定基準**:

| 優先順位 | 対象 | 選定条件 |
|---------|------|---------|
| **1位** | P1（最優先）候補 | まず真っ先に検討 |
| **2位** | P2（高優先）候補 | P1で不十分な場合 |
| **3位** | P3（中優先）以下 | 特定用途で有効な場合 |
| **新規** | 新規発見ツール | 即座に評価（→02_tool_maximization_policy.md 参照） |

**ツール選定フォーマット**:
```
今回の作業タイプ: [調査/比較/実装/修正/検証/記録/GitHub]
選定したツール: 
  - 1位: [ツール名] (理由)
  - 2位: [ツール名] (理由)
使用予定なし: [候補名] (理由)
```

**チェックリスト**:
- [ ] 既存採用候補（D1・D2）の適性確認
- [ ] 新規候補があるか確認
- [ ] 各候補の decision_state 確認（D3以下は不採用判定）
- [ ] 優先順位に基づくツール順序付け

---

### STEP 2: 比較（Comparison Phase）

**目的**: 複数候補がある場合の採用判断

#### 2-1 比較対象選定

**実行内容**:
- phase2_decision_foundation/02_comparison_units_and_matrix_plan.md から該当する比較グループ（CG-01〜CG-06）を特定
- 各候補の compared_with フィールドで過去の比較対象を確認

**比較グループ一覧** (phase2から):
- CG-01: Router/Gateway比較
- CG-02: Orchestration/Workflow比較
- CG-03: Evaluation/Observability比較
- CG-04: Agent Framework比較
- CG-05: MCP Agent/Library比較
- CG-06: Knowledge Management比較

**選定ロジック**:
```
複数候補がある
  ├─ YES → 過去の比較結果があるか確認
  │         ├─ ある → 再評価が必要か確認（信頼区分・鮮度）
  │         └─ ない → 比較実施（→2-2へ）
  └─ NO → 単一候補で進行（→採用判断）
```

**チェックリスト**:
- [ ] 比較対象が2件以上存在する
- [ ] 過去の比較結果を確認
- [ ] 再評価が必要か（鮮度・新規候補出現）確認

#### 2-2 評価基準適用

**実行内容**:
- phase2_decision_foundation/01_evaluation_criteria.md の10軸評価基準（E1〜E10）を適用
- 各候補のスコア差分を定量的に比較

**評価軸**（E1〜E10）:
- E1: 機能スコープ（このタスクに必要な機能を実装しているか）
- E2: 採用実績（導入実績は十分か）
- E3: 活性度（プロジェクト活性度は高いか）
- E4: 運用性（設定・保守の難易度）
- E5: 学習曲線（習得難易度）
- E6: 互換性（既存資産との互換性）
- E7: 安全性（セキュリティ・信頼性）
- E8: パフォーマンス（速度・効率）
- E9: コスト効率（実装・運用コスト）
- E10: 長期メンテナンス性（将来性・継続性）

**比較結果の記録** (phase2/04形式):
```
比較グループ: [CG-XX]
比較対象: [候補A] vs [候補B] vs [候補C]
評価結果:
  候補A: E1=5 E2=5 E3=4 ... (合計スコア: XX)
  候補B: E1=4 E2=4 E3=5 ... (合計スコア: XX)
  候補C: E1=3 E2=3 E3=3 ... (合計スコア: XX)
最有力候補: [候補名]
理由: [定性的な説明]
```

**チェックリスト**:
- [ ] すべての評価軸（E1〜E10）を同一基準で評価
- [ ] スコアの根拠を記録
- [ ] 定量スコアが同点の場合は定性理由を補足

#### 2-3 採用判断

**実行内容**:
- phase2_decision_foundation/05_adoption_build_vs_buy_priority.md に基づき、D1〜D5の判定を確定

**採用状態コード**:
- **D1**: 採用候補（実装予定）
- **D2**: 準採用（条件次第）
- **D3**: 保留（再検討待ち）
- **D4**: 不採用（却下）
- **D5**: 検討対象外

**判定ロジック**:
```
比較スコア1位か？
  ├─ YES → E1（機能スコア）が5か？
  │         ├─ YES → D1（採用）
  │         └─ NO  → D2（準採用）
  └─ NO  → E1が4以上か？
            ├─ YES → D2（準採用）
            └─ NO  → D3/D4（保留・不採用）
```

**記録フォーマット**:
```
候補: [候補ID] [候補名]
判定: [D1/D2/D3/D4/D5]
理由: [定量スコア + 定性理由]
前提条件: [あれば記載、なければ「なし」]
再評価条件: [beta→GA, 新機能, 代替候補出現時など]
```

**チェックリスト**:
- [ ] 前回の判定状態との差分確認
- [ ] 再評価条件の成立有無確認
- [ ] D3（保留）の場合は再検討条件を明記

---

### STEP 3: 実装（Implementation Phase）

**目的**: 採用判断に基づき、実際に機能を組み込む

#### 3-1 対象固定

**実行内容**:
- 今回実装する対象（ファイル・機能・モジュール）を明確にする
- phase1_information_foundation または phase2_decision_foundation から対象を特定

**固定方法**:
```
実装対象: [ファイルパス or 機能名]
  ├─ 新規実装 or 既存修正: [区分]
  ├─ 依存関係: [関連ファイル]
  ├─ 影響範囲: [他ファイルへの波及]
  └─ 完了条件: [何ができたら終わりか]
```

**チェックリスト**:
- [ ] 対象が具体的で、曖昧さがない
- [ ] 依存関係を全て洗い出した
- [ ] 影響範囲を把握した
- [ ] 完了条件を定量的に記述

#### 3-2 ツール固定

**実行内容**:
- STEP 1で選定したツールの中から、今回の実装に使うツール・MCP・API を最終決定
- build_vs_buy_state（B1〜B5）で自作/流用を確定

**ツール確定フォーマット**:
```
使用ツール:
  - 主: [ツール名] (役割: [何をするか])
  - 副: [ツール名] (役割: [何をするか])
自作/流用: [B1: 全面流用 / B2: 大部分流用 / B3: 選別流用 / B4: 自作優先 / B5: 全面自作]
理由: [why this tool/build choice]
```

**build_vs_buy_state の選定基準**:
- **B1**: pip install で全部入り利用可能
- **B2**: OSS を中核に、カスタマイズで対応
- **B3**: 複数OSS を部分採用
- **B4**: OSS の参考実装は参照するが自作主体
- **B5**: 0 から自作

**チェックリスト**:
- [ ] ツールの decision_state が D1 or D2 である（D3以下は不採用）
- [ ] ツールの build_vs_buy_state を確認
- [ ] 自作/流用の根拠をフェーズ2から確認

#### 3-3 実装実行

**実行内容**:
- ツール・MCP・API を実際に組み込む
- コード作成・ファイル作成・設定変更を実施

**実装パターン**（タイプ別）:

| 実装タイプ | 工程 | チェックポイント |
|-----------|------|-----------------|
| **pip install** | install → import → 初期化 | requirements.txt 更新 |
| **git clone** | clone → setup.py install → 初期化 | .gitmodules 設定 |
| **ファイルコピー** | 参考実装の参照実装をコピー → カスタマイズ | オリジナルURL記載 |
| **API呼び出し** | API key設定 → リクエスト実装 → レスポンス処理 | `.env` に key 隔離 |
| **自作実装** | ゼロから実装 → テスト → 動作確認 | phase2 の参考実装参照 |

**実装記録フォーマット**:
```
実装内容:
  - ツール: [ツール名]
  - 追加ファイル: [ファイル一覧]
  - 変更ファイル: [ファイル一覧]
  - 削除ファイル: [ファイル一覧]
  
実装の根拠:
  - phase1 情報源: [A-XX]
  - phase2 比較結果: [CG-XX]
  - 参考実装: [リポジトリ/URL]
```

**チェックリスト**:
- [ ] 実装対象のファイルが全て作成/更新された
- [ ] 依存関係が解決された（import エラーがない）
- [ ] 外部キー・認証情報を `.env` に隔離

#### 3-4 自己検証

**実行内容**:
- 実装直後に以下の検証を実施

**検証の4ステップ**:

```
1. 構文検証
   ├─ Python: python -m py_compile [file]
   ├─ JavaScript: node -c [file] or eslint
   ├─ YAML: yamllint [file]
   └─ JSON: jq . < [file]

2. インポート・依存関係検証
   ├─ Python: python -c "import [module]; print([module].__version__)"
   ├─ Node: npm ls
   └─ Go: go mod verify

3. 機能検証
   ├─ Unit test が成功するか
   ├─ Manual test で想定動作するか
   ├─ エラーハンドリングが機能するか
   └─ Edge case に対応しているか

4. パフォーマンス・セキュリティ検証
   ├─ 処理時間が許容値以内か
   ├─ 認証情報が平文に含まれていないか
   ├─ SQL injection / XSS に対応しているか
   └─ 依存ライブラリの脆弱性がないか (pip audit, npm audit)
```

**検証結果の記録**:
```
自己検証:
  - 構文検証: PASS / FAIL
  - インポート検証: PASS / FAIL
  - 機能検証: PASS / FAIL
  - パフォーマンス検証: PASS / FAIL
問題箇所: [あれば記載]
対応内容: [あれば記載]
```

**チェックリスト**:
- [ ] すべての検証が PASS
- [ ] 問題があれば → STEP 4（修正）へ
- [ ] 問題がなければ → 差分確認へ

---

### STEP 4: 修正（Fix Phase）

**目的**: 自己検証で見つかった問題を修正する

#### 4-1 問題把握

**実行内容**:
- 自己検証で FAIL した項目の詳細を記録

**問題記録フォーマット**:
```
問題ID: [一意に特定できる ID]
問題内容: [何が起こったか]
  - エラーメッセージ: [full stack trace]
  - 再現手順: [どうすれば再現するか]
  - 影響範囲: [どの機能に影響するか]
根本原因推測: [何が原因か]
優先度: [P0: システム停止 / P1: 主機能障害 / P2: 副機能障害 / P3: 軽微]
```

**チェックリスト**:
- [ ] エラーログ全体を記録
- [ ] 再現手順を明記
- [ ] 根本原因仮説を立てる

#### 4-2 修正実行

**実行内容**:
- 根本原因を特定し、コードを修正

**修正方法**:
1. 仮説に基づき修正コードを作成
2. その修正が正しいことを小規模テストで確認
3. その修正が他の機能を壊していないことを確認

**修正記録フォーマット**:
```
問題ID: [前の問題ID]
修正内容:
  - 変更ファイル: [ファイル]
  - 変更行: [行番号]
  - 修正前: [修正前コード]
  - 修正後: [修正後コード]
  - 理由: [なぜこれで直るのか]
```

**チェックリスト**:
- [ ] 修正コードが小規模テストで検証された
- [ ] 修正による副作用がないことを確認
- [ ] コミット前に一度レビュー

#### 4-3 再検証

**実行内容**:
- 修正後に STEP 3-4 の検証を再実行

**再検証フロー**:
```
修正実施
  ↓
自己検証実行（STEP 3-4 と同じ）
  ↓
PASS？
  ├─ YES → 修正完了（→ STEP 5: 検証へ）
  └─ NO  → 新たな問題が見つかった
           ├─ 同じ問題か？ → 修正アプローチを見直し（4-2へ戻る）
           └─ 別の問題か？ → 新規問題として追加（4-1へ）
```

**チェックリスト**:
- [ ] 全検証が PASS
- [ ] 同一問題の再発がない
- [ ] 新たな問題が出ていない

---

### STEP 5: 検証（Verification Phase）

**目的**: 実装全体が完了条件を満たしているか確認

#### 5-1 完了条件確認

**実行内容**:
- STEP 3-1 で定義した「完了条件」をすべてチェック

**完了条件の形式**:
```
完了条件1: [何をできるようになるか]
  - チェック方法: [どうやって確認するか]
  - 実施結果: [PASS / FAIL]
  
完了条件2: ...
```

**典型的な完了条件例**:
- 「[ツール名] が [ファイル名] に正しく import される」→ `python -c "import ..."`
- 「[API] の呼び出しが成功する」→ テストコールで response 確認
- 「[機能] が [条件] で動作する」→ 実装テストで動作確認
- 「エラーハンドリングが実装されている」→ エラー時の動作確認

**チェックリスト**:
- [ ] すべての完了条件が定義されている
- [ ] すべての完了条件を検証した
- [ ] 未達条件はないか

#### 5-2 影響範囲確認

**実行内容**:
- STEP 3-1 で確認した「影響範囲」に実装が悪影響を与えていないか確認

**影響確認フロー**:
```
依存関係のあるファイルを全て特定
  ↓
各ファイルで以下を確認：
  - import 関係はないか？ → あれば import テスト
  - 呼び出し関係はないか？ → あれば機能テスト
  - データ形式の互換性はないか？ → あれば形式テスト
  - 設定ファイル依存はないか？ → あれば設定テスト
  ↓
影響なし → 検証完了
影響あり → 設計上問題か、実装上問題か、根本原因特定 → STEP 4（修正）へ
```

**影響範囲検証記録**:
```
関連ファイル: [ファイル一覧]
各ファイルへの影響:
  - [ファイルA]: [影響内容] → [検証結果]
  - [ファイルB]: [影響内容] → [検証結果]
副作用有無: [あり/なし]
```

**チェックリスト**:
- [ ] すべての依存ファイルを特定した
- [ ] すべての依存ファイルに対して影響テストを実施
- [ ] 副作用がないこと確認

#### 5-3 差分確認

**実行内容**:
- git diff で今回追加/変更/削除したファイルを確認

**差分確認の詳細**:
```
$ git diff --name-status
  M phase4_execution_foundation/01_execution_flow.md
```

### STEP 7: GitHub反映（GitHub Integration）

**目的**: ローカル作業を GitHub に記録し、トレーサビリティを確保

#### 7-1 コミット作成

**実行内容**:
```bash
$ git add [変更ファイル]
$ git commit -m "[コミットメッセージ]"
```

**コミットメッセージ形式**:
```
[type]([scope]): [subject]

[body]

[footer]
```

**type** (以下から選択):
- `feat`: 新機能
- `fix`: バグ修正
- `docs`: ドキュメント
- `refactor`: リファクタリング（機能変更なし）
- `perf`: パフォーマンス改善
- `test`: テスト追加・修正
- `chore`: ビルド・ツール・依存関係変更

**scope** (フェーズ4の場合):
- `phase4`: フェーズ4全体
- `phase4-01`: 実行フロー
- `phase4-02`: ツール最大活用
- など

**subject** (命令形, 50字以内):
- 「...する」形式で記述
- 例: `Add execution flow definition for phase 4`

**body** (詳細説明):
- **What**: 何を変えたか
- **Why**: なぜ変えたか
- **How**: どう変えたか
- 参考情報: phase2の判断根拠・参考実装など

**footer** (参考情報):
```
Relates-to: phase2/CG-04
References: https://github.com/.../issues/123
```

**コミットメッセージ例**:
```
feat(phase4-01): Define execution flow from investigation to GitHub integration

What: Added complete execution flow (7 steps) with tool selection, knowledge reference, and self-verification points.
Why: Phase 4 requires standardized flow to maximize tool usage and ensure quality assurance.
How: Defined flow based on phase 3 knowledge foundation (KL-1 to KL-4) and phase 2 evaluation results.

Relates-to: phase3_knowledge_foundation/08_phase4_handoff.md
Relates-to: phase2_decision_foundation/04_candidate_evaluation.md
```

**チェックリスト**:
- [ ] コミットメッセージが具体的である
- [ ] type・scope が正確である
- [ ] body に何を・なぜ・どうが記載されている
- [ ] 参考情報がある場合は footer に記載

#### 7-2 PR作成（必要時）

**実行内容**:
- 重要な変更・レビューが必要な変更の場合、PR を作成

**PR作成タイミング**:
- 複数ファイルの連携が複雑な場合
- 重要な判断が含まれている場合
- チームレビューが必要な場合

**PR説明テンプレート**:
```markdown
## Summary
[何を変えたか、なぜか]

## Changes
- [変更内容1]
- [変更内容2]

## Related to
- phase2_decision_foundation/[該当ファイル]
- phase3_knowledge_foundation/[該当ファイル]

## Verification
- [x] Self-verification passed
- [x] Impact range checked
- [x] Diff reviewed
```

**チェックリスト**:
- [ ] PR タイトルが具体的である
- [ ] Summary が読みやすい
- [ ] Changes セクションで差分を簡潔に説明
- [ ] 関連ファイル・参考情報が明記されている

#### 7-3 origin/main へ push

**実行内容**:
```bash

$ git push origin [branch-name]
  (GitHub でPRを作成・マージ)

# ローカルコミットの場合

```

**push 前チェックリスト**:
- [ ] 全変更がコミットされている
- [ ] コミットメッセージが適切である
- [ ] git log で履歴が正しいか確認
- [ ] origin/main と競合がないか確認

**push 後チェックリスト**:
- [ ] GitHub の main ブランチに反映されているか確認
- [ ] CI/CD パイプライン（あれば）が成功しているか確認
