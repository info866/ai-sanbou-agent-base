# ツール最大活用方針（Tool Maximization Policy）

> **ステップ対応**: フェーズ4作業指示書 ブロック2  
> **最終更新**: 2026-04-13

---

## この文書の役割

フェーズ2で採用されたツール・MCP・API（D1・D2候補、P1・P2優先度）を、限定的ではなく「最大限」活用するための原則・具体的な使用場面・ツール組み合わせを定義する。

---

## 原則1：既存採用候補は主戦力

### 方針

フェーズ2で D1（採用候補）と判定された 10 件のツール・MCP・API は、フェーズ4では原則として以下を満たす：

1. **最初に検討する**: 新規ツールを探す前に、まずこれらを使えないか確認
2. **積極的に組み合わせる**: 単独では足りなくても、複数組み合わせで対応できないか検討
3. **深掘り利用する**: 基本的な使い方だけでなく、高度な機能・応用パターンを活用
4. **結論は「使わない」に決めるな**: 「使う前提で工夫する」という思考から始める

### D1 採用候補の全リスト（10件）

| item_id | 候補名 | 役割 | 深掘りポイント |
|---------|--------|------|--------------|
| **F-002** | **MCP** | 全フェーズの接続基盤 | MCP Registry API、Server実装・デバッグ、カスタムMCPサーバー作成 |
| **F-003** | **Subagents** | エージェント並列実行・スケーリング | 16フロントマターフィールドの全活用、条件分岐・例外処理 |
| **F-004** | **Hooks** | イベント駆動の自動化 | HTTPフック・Agentフック・ファイル変更検出、複合条件トリガー |
| **F-005** | **Skills** | ノウハウのパッケージ化・再利用 | supporting files、依存関係管理、スキル間呼び出し |
| **F-009** | **Claude Agent SDK** | 本番エージェント構築基盤 | ワークフロー定義、状態管理、エラーハンドリング、並列実行 |
| **F-013** | **anthropics/skills** | スキル資産の宝庫 | 利用可能なスキルの棚卸し、既存スキルの応用パターン探索 |
| **F-015** | **claude-agent-sdk-python** | Python本番実装 | LiteLLM・promptfooとの統合、複雑な状態遷移実装 |
| **F-019** | **MCP Servers** | リファレンス実装集 | 実装パターン学習、セキュリティベストプラクティス |
| **F-025** | **Memory Tool** | クロスセッション記憶 | 会話履歴・判断記録・学習データの自動保存、検索・参照 |
| **F-032** | **promptfoo** | LLM評価フレームワーク | YAML宣言的設定、マルチモデル比較、メトリクス自動計算 |

## 原則2：P1・P2候補は即座に活用

### 方針

P1（最優先）6件・P2（高優先）6件の合計 12 件は、以下の優先度で使用する：

**P1（最優先）**: 次のタスク開始時に真っ先に検討し、使えるなら即座に組み込む
**P2（高優先）**: P1で不十分な場合、または並列して活用

### P1・P2 の全リスト（12件）

| 優先度 | item_id | 候補名 | 使用タイミング |

| **P1** | F-005 | Skills | ノウハウのパッケージ化・再利用が必要なとき |
| **P1** | F-009 | Claude Agent SDK | 本格的なエージェント実装のたびに |

| **P2** | F-032 | promptfoo | LLM比較・評価・メトリクス自動計算 |

## 原則3：D2（試験導入）候補は段階的に組み込む

### 方針

フェーズ2で D2（試験導入候補）と判定された 12 件は、以下の方針で段階的に活用：

1. **小規模で試す**: いきなり本格導入ではなく、1タスク・1ワークフローで試験
2. **効果を実感する**: 試験で得た成果・問題点を記録
3. **本格導入を判定**: 効果があるなら本格導入へ移行、問題があるなら改善または保留

### D2 試験導入の段階的計画

| item_id | 候補名 | 試験内容 | 試験開始時期 | 本格導入判定 |

| F-010 | GitHub Actions for Claude Code | 1つのPR自動レビュー設定 | Phase4第2週 | 効果あり→全PRに展開 |
| F-011 | Cloud Scheduled Tasks | 1つの定期タスク自動実行 | Phase4第2週 | 効果あり→複数タスクに展開 |
| F-014 | anthropics/claude-code-action | GitHub Actions YAMLテスト | Phase4第2週 | F-010と同時実施 |

| F-018 | MCP Registry | MCPサーバー選定で1度使用 | Phase4第1週 | APIが使えるか確認→本格導入 |

## 原則4：P3以下・D3以上の候補の明示的な「使わない」判定

### 方針

以下の候補は「今は使わない」ことを明示的に定める（後回し・検討対象外ではなく）：

- **D3（保留）**: 価値はあるが時期尚早→フェーズ4では使わない（フェーズ4終了時に再評価）
- **D4（不採用）**: 現時点では使わない（再検討条件成立まで検討対象外）
- **D5（監視継続）**: 技術トレンド監視のみ。フェーズ4では使わない
- **P3〜P5**: 優先度低い→フェーズ4では使わない（余裕があれば評価）
- **B4・B5**: 自作優先または判断保留→フェーズ4では使わない

### 明示的に「使わない」判定の記録方法

ファイル内で、各タスク・作業ブロックごとに以下を記述：

```
## [タスク名]

### 使用ツール
- [P1候補]: [具体的な使用方法]
- [P2候補]: [具体的な使用方法]

### 使用しないツール（明示的判定）
- [D3/D4/P3候補]: 理由「[保留理由]」→フェーズ4では不採用
- [候補名]: 理由「[理由]」→フェーズ4では不採用

### 根拠（なぜ使わないのか）
[フェーズ2の判定をここに引用して根拠を明示]
```

## 原則5：複数ツールの活用シナリオ

### シナリオ1：エージェント実装（MCP + Subagents + Claude Agent SDK + Skills）

**タスク**: 複雑なエージェントワークフローを実装

**使用ツールの組み合わせ**:
1. **Claude Agent SDK** (F-009): コアのエージェント実装基盤
2. **MCP** (F-002): 外部リソース接続
3. **Subagents** (F-003): 複数エージェントの並列実行・スケーリング
4. **Skills** (F-005): ノウハウのパッケージ化

**実装フロー**:
```
Agent SDK + MCP
  ↓ (複数エージェントが必要)
  + Subagents
  ↓ (ノウハウを再利用したい)
  + Skills
  ↓ (完成)
```

**活用ポイント**:
- MCPなしに外部連携は実現困難→必ず使う
- Subagentsで単一エージェントの限界を超える
- Skillsでコード再利用性を最大化

---

### シナリオ2：自動化パイプライン（Hooks + GitHub Actions + Cloud Scheduled Tasks）

**タスク**: ファイル変更・GitHub操作・定期タスクの自動実行

**使用ツールの組み合わせ**:
1. **Hooks** (F-004): ローカルイベント駆動
2. **GitHub Actions** (F-010 + F-014): GitHub内での自動化
3. **Cloud Scheduled Tasks** (F-011): 定期実行

**自動化フロー**:
```
ファイル変更・push
  ↓ (Hooks で検出)
  → GitHub Actions (自動テスト・PR自動レビュー)
  → Cloud Scheduled Tasks (定期メンテナンス)
```

**活用ポイント**:
- 3つの層で異なるレベルの自動化を実現
- Hooksで即座に反応、GitHub Actionsで統合、Scheduled Tasksで定期化

---

### シナリオ3：LLM評価パイプライン（promptfoo + LiteLLM + claude-agent-sdk-python）

**タスク**: 複数モデル・複数プロンプトの比較評価

**使用ツールの組み合わせ**:
1. **promptfoo** (F-032): 評価フレームワーク
2. **LiteLLM** (F-022): マルチモデルAPI統一
3. **claude-agent-sdk-python** (F-015): Python実装

**評価パイプライン**:
```
promptfoo YAML設定（複数プロンプト・モデル定義）
  ↓
LiteLLM（複数モデルへのAPI統一）
  ↓
claude-agent-sdk-python（エージェント実装のコンテキストで評価）
  ↓
結果分析・採用判定
```

**活用ポイント**:
- promptfooで宣言的に比較条件定義
- LiteLLMでモデル切替を簡素化
- Agent SDK contextで実装パターンの評価も可能

## 実装レベル：各作業単位でのツール最大活用

### 調査フェーズでのツール活用

```
ツール選定:
  ├─ 既知知識: phase3_knowledge_foundation から参照
  │  └─ 使用ツール: WebFetch（ドキュメント取得）
  │
  ├─ 公式情報確認: リリースノート・ドキュメント
  │  └─ 使用ツール: WebSearch（最新情報）+ WebFetch（公式情報）
  │
  └─ GitHub確認: OSS活性度・代替候補探索
     └─ 使用ツール: GitHub API（stars, 更新日, issue）
```

**D1・D2候補の中から選定**:
- P1: 必ず検討
- P2: P1で不足なら追加検討
- D2: 小さく試す対象として検討

## チェックリスト：「最大限活用」の実施確認

各タスク・作業ブロック完了時に、以下を確認：

### 候補検討の網羅性
- [ ] P1（最優先）6件を全て検討した（「使えるか」「組み合わせられるか」を確認）
- [ ] P2（高優先）6件を全て検討した（P1で不足なら追加検討）
- [ ] D2（試験導入）候補で該当するものを試験検討した

### 組み合わせ検討
- [ ] 複数ツールの組み合わせで効果向上できないか検討した
- [ ] 単独では足りないか、組み合わせで対応できないか確認した

### 深掘り利用
- [ ] 単なる「基本的な使い方」に留まっていないか
- [ ] ツールの高度な機能・応用パターン・オプションを活用したか

### 明示的な「使わない」判定
- [ ] D3・D4・P3以下の候補は「今は使わない」と明示的に記録した
- [ ] その理由（フェーズ2の判定への参照）を記載した

### ツール効果の記録
- [ ] 各ツール使用の効果・課題を記録した
- [ ] 効果があれば D2→本格導入への推奨を記録
