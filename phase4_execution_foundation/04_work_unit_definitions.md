# 作業単位定義（Work Unit Definitions）

> **ステップ対応**: フェーズ4作業指示書 ブロック4  
> **最終更新**: 2026-04-13

---

## この文書の役割

フェーズ4で実行する調査・比較・実装・修正・記録各タスクを「単位化」し、各単位の開始条件・終了条件・チェックポイントを定義。

---

## 単位1：調査単位

### 定義

「1つの候補について、既知知識参照から公式情報確認までを完了するまで」

### 開始条件
- 対象候補が明確に決定している
- phase2_decision_foundation/05_adoption_build_vs_buy_priority.md で採用状態が確認されている
- 今回のタスクで何を知る必要があるかが明確になっている

### 実行フロー

```
1. phase3_knowledge_foundation から既知知識参照
   ├─ trust_level 確認
   ├─ last_verified_at 確認
   └─ recheck_required_flag 確認

2. 再確認ルール（C1〜C4）に基づき公式情報確認
   ├─ beta/GA 状態確認（リリースノート）
   ├─ API 仕様変更有無（公式ドキュメント）
   ├─ Pricing 情報確認（公式ページ）
   └─ OSS 活性度確認（GitHub）

3. ツール選定
   ├─ P1/P2 候補の適性確認
   ├─ 新規候補の有無確認
   └─ 使用予定ツール決定
```

### 終了条件
- [ ] 既知知識（trust_level）を確認した
- [ ] 必要な公式情報を確認した
- [ ] 対象候補に対して使用するツール・MCP・API が明確に決まった
- [ ] 次ステップ（比較 or 実装）に進む判断ができている

### チェックポイント

| 項目 | 確認内容 |
|-----|--------|
| **知識鮮度** | last_verified_at が 90日以内か、または T1 確定知識か |
| **信頼区分** | T1〜T4 を確認し、必要な再確認を実施したか |
| **ツール選定** | P1 優先で候補を検討したか |
| **根拠明確性** | なぜこの候補を選んだか説明できるか |

## 単位2：比較単位

### 定義

「2〜3の候補を同一基準で定量評価し、採用判断に至るまで」

### 開始条件
- 比較対象候補が 2件以上決定している
- phase2_decision_foundation/01_evaluation_criteria.md の評価基準（E1〜E10）が利用可能である
- 今回の比較の「決定対象」が明確（採用判定 or 優先順位決定 or 効果検証）

### 実行フロー

```
1. 比較対象選定
   ├─ phase2_decision_foundation/02_comparison_units_and_matrix_plan.md から該当する比較グループ確認
   └─ 過去の比較結果があるか確認（重複比較は避ける）

2. 評価基準適用（E1〜E10）
   ├─ 各候補の評価スコアを同一基準で採点
   ├─ スコアの根拠を記録
   └─ 定量スコアが同点の場合は定性理由を補足

3. 採用判断
   ├─ スコアの高い順に D1〜D5 を判定
   └─ 再評価条件・前提条件があれば記録
```

### 終了条件
- [ ] すべての比較対象（2件以上）を評価基準で採点した
- [ ] 採用状態（D1〜D5）が確定した
- [ ] 前回との判定変更があれば、その理由を記録した
- [ ] 再評価条件が明記されている（保留・不採用の場合）

### チェックポイント

| 項目 | 確認内容 |
|-----|--------|
| **基準統一** | E1〜E10 すべてを同一基準で評価したか |
| **定量性** | スコアが数値で記録されているか |
| **根拠** | 「なぜこのスコアか」説明できるか |
| **判定明確性** | D1〜D5 のいずれかが確定しているか |

## 単位3：実装単位

### 定義

「対象ツール確定から git diff で差分確認まで」＝「1つのツール導入の完全サイクル」

### 開始条件
- 使用するツール・MCP・API が確定している
- 自作/流用判断（B1〜B5）が明確である
- 実装の完了条件が定義されている

### 実行フロー

```
1. 対象固定
   ├─ 実装ファイル・機能を明確にする
   ├─ 依存関係を洗い出す
   └─ 完了条件を定量的に記述

2. ツール固定
   ├─ build_vs_buy_state 確認（B1〜B5）
   ├─ 主ツール・副ツール選定
   └─ 自作/流用の理由確認

3. 実装実行
   ├─ install / import / setup
   ├─ コード作成 / ファイル生成
   └─ 基本的な動作確認

4. 自己検証
   ├─ 構文検証（python -m py_compile など）
   ├─ インポート検証
   ├─ 機能検証（unit test / manual test）
   └─ パフォーマンス・セキュリティ検証

5. 差分確認
   ├─ git status で全変更ファイル確認
   ├─ git diff で内容確認
   └─ 意図通りの変更かチェック
```

### 終了条件
- [ ] 実装ファイルが全て作成/更新された
- [ ] 依存関係が解決された（import エラーなし）
- [ ] 自己検証（構文・機能・セキュリティ）が PASS した
- [ ] git diff で変更が確認できる
- [ ] 完了条件がすべて満たされている

### チェックポイント

| 項目 | 確認内容 |
|-----|--------|
| **対象明確性** | 実装対象・依存関係・完了条件が曖昧でないか |
| **自動化度** | B1（全面流用）か、カスタマイズが必要か確認したか |
| **検証完全性** | 構文・機能・パフォーマンス・セキュリティ全て確認したか |
| **差分確認** | git diff で意図通りの変更か確認したか |

## 単位4：修正単位

### 定義

「自己検証で見つかった問題から、再検証まで」＝「問題検出→修正→確認の完全サイクル」

### 開始条件
- 自己検証で FAIL が見つかっている
- 問題内容が具体的に記録されている
- 修正対象が明確である

### 実行フロー

```
1. 問題把握
   ├─ エラーメッセージ全体を記録
   ├─ 再現手順を明記
   └─ 根本原因仮説を立てる

2. 修正実行
   ├─ 仮説に基づき修正コード作成
   ├─ 小規模テストで検証
   └─ 副作用がないこと確認

3. 再検証
   ├─ 修正前の検証項目を全て再実行
   ├─ 同一問題の再発がないか確認
   └─ 新たな問題が出ていないか確認
```

### 終了条件
- [ ] 根本原因が特定されている
- [ ] 修正コードが実装されている
- [ ] 再検証で PASS している
- [ ] 新たな問題が出ていない
- [ ] 必要に応じて修正内容を git diff で確認した

### チェックポイント

| 項目 | 確認内容 |
|-----|--------|
| **根本原因** | 表面的な症状ではなく根本原因に対応しているか |
| **副作用** | 修正による他機能への影響を確認したか |
| **再発防止** | 同類問題が他にないか確認したか |
| **完全性** | 修正前の検証項目を全て再実行したか |

## 単位5：記録単位

### 定義

「1つの作業ブロック完了時に、成果・判断根拠・課題・次アクションを全て記録する」

### 実行フロー

```
1. 使用ツール記録
   ├─ 使用したツール・MCP・API を全て列挙
   ├─ 各ツールの役割を明記
   └─ 採用判定への参照を記載

2. 変更内容記録
   ├─ ファイル単位で「追加/修正/削除」を記録
   ├─ 各変更の理由を記載
   └─ git diff で確認できる形式

3. 検証方法記録
   ├─ 実施した検証項目と結果を記録
   ├─ 問題があった場合は対応内容を記載
   └─ 再検証結果を記録

4. 未完了記録
   ├─ スコープ外のこと・後回しになったことを列挙
   ├─ 理由を記載
   └─ 対応予定時期を記載

5. 次アクション記録
   ├─ 次にやるべきことを明記
   ├─ 優先順位をつける
   └─ 依存関係を記載
```

### 終了条件
- [ ] 使用ツールが全て記録されている
- [ ] 変更内容がファイル単位で記録されている
- [ ] 実施した検証項目と結果が全て記録されている
- [ ] 問題があった場合、対応内容が記載されている
- [ ] 未完了事項が明記されている
- [ ] 次アクションが具体的に記載されている

### チェックポイント

| 項目 | 確認内容 |
|-----|--------|
| **ツール追跡性** | なぜこのツールを選んだか追跡できるか |
| **変更追跡性** | 何をどう変えたか、なぜ変えたか記載されているか |
| **検証完全性** | 全検証項目が記録されているか |
| **透明性** | スコープ外・後回し・課題が隠れていないか |

## 単位ごとの並列実行可能性

### 並列可能な単位

以下の単位は並列実行可能（依存関係なし）：

```
複数候補の調査 → 各候補で独立した調査単位を並列実行
複数候補の実装 → 依存関係がなければ並列実行
新規ツール評価 → 複数ツールを同時に試験導入可能
```

### 並列実行時の注意

- **ツール競合**: 同じツール選定枠で複数候補が競合する場合は比較単位を挟む
- **ファイル競合**: 同じファイルの修正が必要な場合は順序付けする
- **リソース競合**: セットアップ・テスト環境が共有の場合は調整
