## フェーズ4の目的（正本から）

1. **実作業をツール中心のフローへ置き換える**
2. **使えるツールを最大限活用して、効率と品質を同時に最大化する**
3. **調査・比較・実装・修正・検証・記録・GitHub反映を一つの標準フローにする**
4. **フェーズ5でAI参謀がそのまま使える実行基盤を作る**

---

## 実行ループ

### Phase 4 を完了するまで以下を繰り返す：

```
1. 作業ブロック実行
   ├─ 実行フロー定義
   ├─ ツール最大活用方針
   ├─ 新規ツール即時投入ルール
   ├─ 作業単位定義
   ├─ 品質保証ルール
   ├─ GitHub接続方針
   └─ フェーズ5接続メモ

2. 正本再読
   ├─ 9.フェーズ4 要件定義書.md を再読
   ├─ 現在地を特定
   └─ 完了条件の達成状況を確認

3. 完全監査
   ├─ 6つの完了条件に対して全て検証
   ├─ 7つの必須成果物の充足状況確認
   ├─ 命名ドリフト・正本ドリフト・古い記述の有無確認
   └─ 各成果物の相互一貫性確認

4. ギャップ洗い出し
   ├─ 未充足の完了条件をリスト化
   ├─ 不完全な成果物をリスト化
   ├─ リポジトリ内の根拠をリスト化
   └─ 残問題を全て明記

5. 根本原因対策
   ├─ ギャップの根本原因を特定
   ├─ 設計レベルの修正
   ├─ 実装レベルの修正
   └─ ファイル・コード更新

6. 正本再読と再監査
   ├─ 修正後に9.フェーズ4 要件定義書.md を再読
   ├─ 修正内容が正本と合致していることを確認
   └─ 全監査項目を再実行
```

**繰り返しの停止条件**: すべての完了条件が満たされ、すべての成果物が存在し、一貫性が確認されたとき。

## 監査と検証

### 完了条件の全検証

各ブロック完了後に以下を実行：

```
1. 正本再読
   ├─ 9.フェーズ4 要件定義書.md パート5「実現すべき状態」を読む
   └─ 現在地を「どの完了条件をいくつ満たしているか」で確認

2. 完全監査
   完了条件6つを全て対象に：
   ├─ 条件1: ツール最大活用が成果物02で定義されているか
   ├─ 条件2: 新規ツール即時投入ルールが成果物03で定義されているか
   ├─ 条件3: 作業単位フローが成果物04で定義されているか
   ├─ 条件4: 自己検証が成果物05で必須化されているか
   ├─ 条件5: GitHub接続が成果物06で組み込まれているか
   └─ 条件6: フェーズ5接続が成果物07で明記されているか

3. ドリフト検査
   ├─ 命名ドリフト: ファイル名・用語の統一性確認
   ├─ 正本ドリフト: 9.フェーズ4 要件定義書との整合性確認
   ├─ 古い記述: 「たぶん」「あとで」「TBD」の有無確認
   └─ 相互一貫性: 成果物7つが相互に矛盾していないか確認

4. リポジトリ根拠確認
   ├─ すべての成果物がphase4_execution_foundation/以下に存在するか
   ├─ ファイルが git status で追跡されているか
   ├─ 不完全なファイル（プレースホルダー・スケルトン）がないか
   └─ すべてのファイルが origin/main と一致しているか
```

## ギャップ洗い出しと修正

### ギャップが見つかった場合

```
1. ギャップをリスト化
   ├─ 不完全な完了条件
   ├─ 不完全な成果物
   ├─ ドリフトしている記述
   └─ リポジトリ内の根拠がない項目

2. 根本原因を特定
   ├─ 設計漏れ（フェーズ4要件定義書と現状の乖離）
   ├─ 実装漏れ（設計は正しいが実装がない）
   ├─ ドリフト（設計は古いが実装が先に進んだ）
   └─ 参照漏れ（必要な入力資産を見落とした）

3. 根本原因対策を実施
   ├─ 設計漏れ: 正本を読み直し、成果物の要件を追加
   ├─ 実装漏れ: 漏れたファイルを実装する
   ├─ ドリフト: 正本との整合を修正する
   └─ 参照漏れ: 入力資産を再確認し、漏れた項目を反映

4. 修正実施と再監査
   └─ 各成果物を修正後、完全監査を全て再実行
```
