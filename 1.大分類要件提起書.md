# 【大分類要件提示書 / タスク名】
お前が今回扱うプロジェクトは、**最新のAI機能・MCP・API・OSS・運用知識を継続的に収集し、評価し、採用判断し、実務へ最適投入できるAI参謀エージェント基盤を段階的に構築するプロジェクト**である。

この文書は、全体計画を固定するための**大分類要件提示書**である。  
ここではフェーズ全体の骨格を定義する。  
個別フェーズの詳細設計・個別実装・個別作業手順は、後続のフェーズ別要件提示書および作業指示書で定義する。  
この文書だけで勝手に詳細実装へ踏み込むな。

---

# 【最重要目的】
このプロジェクトの最終目的は、以下を満たすことである。

1. **新しく公開・更新されるAI関連機能を継続的に把握できること**
2. **それらの機能・MCP・API・OSS・既存エージェントを評価できること**
3. **自作すべきか、既存資産を採用すべきか、根拠付きで判断できること**
4. **採用した能力を、実務に投入可能な形で管理・接続・運用できること**
5. **最終的に、ユーザーの代わりに能力活用を最適化するAI参謀エージェントへ到達すること**

---

# 【この大分類要件提示書の役割】
この文書の役割は以下である。

- プロジェクト全体のゴールを固定する

# 【前提認識】
このプロジェクトでは、以下を前提として扱うこと。

## 1. 問題の本質
ユーザーの本当の課題は、単なるコーディング不足ではない。  
本質は、**Claude Code / Anthropic / OpenAI / MCP / API / OSS / 既存エージェントの更新速度と複雑さに、人間単独では追従しきれないこと**である。

## 2. 作るべきものの本質
作るべきものは、単なるチャットボットではない。  
単なるRAGでもない。  
単なるモデルルーターでもない。  
作るべきものは、**能力の追跡・評価・採用・実行投入を担うAI参謀エージェント基盤**である。

## 3. 開発方針
このプロジェクトは一気通貫で完成させるのではなく、  
**大分類 → フェーズ単位の要件提示書 → フェーズ単位の作業指示書**  
の順で分割し、段階的に確実に構築する。

## 4. 採用方針
世の中に既に優れたOSS・既存エージェント・既存実装があるなら、  
それを見落としてゼロから自作することは原則として避ける。  
ただし、既存資産の採用は無条件ではなく、必ず評価と比較を伴う。

# 【全体ゴール】
最終的に到達すべき状態を以下に定義する。

## ゴールG1｜継続情報収集
公式公開情報、GitHub公開情報、MCP関連情報、既存OSS、実装例、運用知見を継続的に収集できること。

## ゴールG2｜能力カタログ化
収集した情報を、単なる記事一覧ではなく、  
「何ができるか」「何に効くか」「何と接続できるか」「どう使うべきか」という観点で構造化管理できること。

## ゴールG3｜採用判断
新機能や既存OSSを見つけたときに、  
それを採用すべきか、見送るべきか、自作すべきかを、根拠付きで判断できること。

## ゴールG4｜知識検索・参照
蓄積された知識や評価結果を、後から必要に応じて検索・参照・要約できること。

## ゴールG5｜実行投入
採用済みの能力を、実務用のエージェント群・MCP・API・作業フローへ接続できること。

## ゴールG6｜AI参謀化
最終的に、ユーザーの代わりに  
「今回は何を使うべきか」  
「既存資産を流用すべきか」  
「どの構成で作業すべきか」  
を判断し、提案し、作業を前進させるAI参謀として機能すること。

# 【大分類フェーズ構成】
このプロジェクトは以下の大分類フェーズで構成する。

---

## フェーズ1｜情報基盤フェーズ
### 目的
新機能・MCP・API・OSS・既存エージェント・公式情報を収集・保存・更新管理できる基盤を構築する。

### 主責務
- 情報源の一覧化
- 収集対象の定義
- 情報の保存先設計
- 収集フロー設計
- 更新追跡の基本構造設計
- 採用前候補の管理構造設計

### 到達状態
- 「どこを見ればよいか」が固定されている
- 情報を保存・整理・分類する基盤がある
- 新情報が入ってきたときに追加管理できる
- 採用候補・保留候補・不採用候補を管理できる

### この段階でまだやらないこと
- 本格的な採用判断ロジックの完成
- 実務投入用のエージェント実装
- 全自動オーケストレーション
- 完成版AI参謀の構築

---

## フェーズ2｜判断基盤フェーズ
### 目的
集めた情報を評価し、使う価値があるか、採用すべきか、自作すべきかを判断できる構造を作る。

### 主責務
- 評価基準の定義
- 採用判断のルール設計
- 自作 vs 流用 vs 保留の比較構造
- リスク評価
- 運用価値評価
- 将来拡張性評価

### 到達状態
- 新機能やOSSを見つけたときに、採用可否を説明できる
- 既存資産を使うか、自作すべきかを比較できる
- 判断結果が記録され、再利用できる
- 感覚ではなく根拠で選定できる

### この段階でまだやらないこと
- 実務エージェント群の本格構築
- 本番運用への全面投入
- 最終オーケストレーター完成

---

## フェーズ3｜知識活用基盤フェーズ
### 目的
蓄積された知識・評価結果・候補情報・採用結果を後から検索・参照・要約しやすくする。

### 主責務
- 検索基盤設計
- 知識参照設計
- 将来的なRAG導入判断
- 文書・構造データの扱い方整理
- 情報取得と活用の接続設計

### 到達状態
- 必要情報を後から引ける
- 同じ調査を何度もやり直さなくて済む
- 収集済み情報が死蔵されない
- 参謀エージェントが知識を再利用できる

### この段階でまだやらないこと
- 全自律型の完成版作業エージェント
- 完全自動実行パイプライン

---

## フェーズ4｜実行基盤フェーズ
### 目的
採用済みの能力を実際の作業系エージェント・MCP・API・作業環境へ接続し、実務で使える形にする。

### 主責務
- 実行アーキテクチャ設計
- エージェント役割分担設計
- MCP / API 接続設計
- 既存OSS組み込み設計
- 実行フロー設計

### 到達状態
- 採用済み能力を作業環境へ投入できる
- 実務用の作業フローが形成される
- 作業系エージェントが参照基盤と接続される
- 参謀判断と実行判断が接続される

---

## フェーズ5｜AI参謀完成フェーズ
### 目的
能力収集・評価・知識活用・実行投入が統合され、ユーザーの代わりに能力活用を最適化するAI参謀エージェントへ到達する。

### 主責務
- 参謀エージェント統合
- 作業前判断の自動化
- 採用候補提示の自動化
- 実務向け提案生成
- 継続改善サイクルの組み込み

### 到達状態
- 今回の作業に何を使うべきか提案できる
- 既存OSSを流用すべきか判定できる
- 新機能を見つけて採用候補化できる
- 実行側へ適切に橋渡しできる
- 人間の勘依存を大幅に減らせる

# 【共通要件】
全フェーズを通じて満たすべき共通要件を定義する。

## 要件R1｜継続更新対応
新機能・更新・既存OSSの変化に追従できる構造であること。

## 要件R2｜ゼロベース自作の抑制
既存資産で十分なものを見落としてゼロから再発明しないこと。

## 要件R3｜評価根拠の保持
採用・不採用・保留の判断には、根拠が残ること。

## 要件R4｜再利用性
一度調査・評価した情報は、後から再利用できること。

## 要件R5｜段階構築
一気に全部作らず、フェーズごとに明確な入口と出口を持つこと。

## 要件R6｜実務接続性
最終的に実際の作業へ投入できる方向で構築されること。  
単なる知識保管庫で終わってはならない。

## 要件R7｜拡張性
今後追加される新機能・新MCP・新API・新OSSを後から取り込めること。

## 要件R8｜安全性
採用判断・接続判断・実行判断において、無制限な自律採用を前提にしないこと。  
評価・確認・停止線を持つこと。

# 【フェーズ進行ルール】
今後の進め方は必ず以下に従うこと。

1. 大分類要件提示書を先に固定する
2. その後、大分類作業指示書を作る
3. その後、フェーズ1要件提示書を作る
4. その後、フェーズ1作業指示書を作る
5. フェーズ1完了後、大分類要件提示書に立ち返る
6. 必要な見直しを行ったうえで、フェーズ2要件提示書へ進む
7. 同様に、各フェーズごとに要件提示書 → 作業指示書 → 実施 → 見直し の順で進める
8. 全体像を見失ったまま局所最適に入らないこと
