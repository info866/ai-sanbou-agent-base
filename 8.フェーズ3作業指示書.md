# 【お前の基本動作原則】
以下を全工程で守れ。

## 原則P1｜正本遵守
フェーズ3要件定義書を最上位の作業正本として扱え。  
迷ったら、勝手な最適化ではなくフェーズ3要件へ戻れ。

## 原則P2｜保存知識を過信しない
知識基盤に入っているものを、そのまま現在の真実として断定するな。  
鮮度が重要なものは必ず再確認条件を持たせろ。

## 原則P3｜構造化優先と文書活用を分ける
状態、優先順位、判断結果、再評価条件のような明示項目は構造化で扱え。  
長文説明、比較メモ、README要約、公式解説は文書知識として扱え。

## 原則P4｜RAGは補助
RAGを導入する場合でも、全知識を一括でそこへ押し込むな。  
RAGは文書知識の検索補助であり、状態管理や判断確定情報の正本ではない。

## 原則P5｜根拠追跡性維持
どの知識が、どの情報源・どの比較・どの判断に基づいているか追跡できるようにしろ。

## 原則P6｜GitHub正本維持
コード、README、設計メモ、差分、変更履歴、比較対象のリポジトリ情報は GitHub を根拠基盤として扱え。  
口頭説明だけで済ませるな。

## 原則P7｜再利用優先
一度作った比較・判断・要約を、後から再検索・再利用できるようにすることを優先しろ。

## ステップ6｜知識活用フローの定義
### 目的
問いが来たときに、どの順で知識を見て、いつ外部再確認するかを固定する。

### やること
最低限、以下の流れを定義しろ。

1. 問いの種類を判定する
2. 構造化メタデータを引く
3. 必要なら判断記録を引く
4. 必要なら文書知識を引く
5. 鮮度が怪しければ再確認判定を行う
6. 必要時のみ公式 or GitHub 再調査へ進む
7. 出力を「確定 / 判断済み / 再確認必要 / 未解決」に分けて返せるようにする

### 完了条件
- 知識をどう使うかの順番が固定されている
- 後続エージェントが流れを再利用できる

### やってはいけないこと
- 問いごとに場当たりで知識源を選ぶこと
- 古い知識をそのまま返すこと

## ステップ8｜再確認ルールの定義
### 目的
古くなりうる知識を安全に扱うための条件を固定する。

### やること
最低限、以下を定義しろ。

#### 8-1 鮮度依存再確認
- 最新機能
- 仕様変更
- beta / GA 状態
- モデル / 制限 / pricing / current status
- 活発なOSSの現況

#### 8-2 重要判断前再確認
- 採用判断に進む前
- 実装投入判断に進む前
- 監督ロジックに組み込む前

#### 8-3 根拠不足再確認
- 比較が少ない
- 情報源が弱い
- 断片情報しかない
- 最終確認日が古い

#### 8-4 RAG単独断定禁止
- RAG結果だけで最新事実を断定しない
- 必要に応じて公式 / GitHubを取りに行く

### 完了条件
- 再確認が必要な知識の条件が定義されている
- 古い知識を誤用しにくい構造になっている

### やってはいけないこと
- 全部毎回再確認する設計にして効率を殺すこと
- 逆に全部再確認不要にすること
