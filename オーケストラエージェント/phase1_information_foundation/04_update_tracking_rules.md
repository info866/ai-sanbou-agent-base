## パート2｜更新差分の記録方法

### 更新履歴テーブルの構造

更新履歴は `02_candidate_catalog.md` の末尾または別ファイル `06_update_log.md`（フェーズ2で作成）に蓄積する。1件の更新に対して1行を追記する。

| フィールド | 型 | 説明 |
|-----------|-----|------|
| update_id | string | UL-001形式の連番ID |
| item_id | string | 対象候補のitem_id（F-001等）または情報源のsource_id（A-001等） |
| detected_at | date | 変化を検知した日付（YYYY-MM-DD） |
| update_source | URL | 変化を確認した情報源URL（Release NotesのURLやGitHubコミットURL等） |
| update_summary | text | 何が変わったかの1〜2文での要約 |
| breaking_change_flag | bool | true = 既存実装に影響する可能性があるbreaking change |
| followup_required_flag | bool | true = 追加調査・実装対応等のフォローアップが必要 |
| notes | text | 補足・対応内容・判断理由 |

**記録例:**
```
| UL-001 | F-012 | 2026-04-20 | https://platform.claude.com/docs/en/release-notes/overview | Managed Agents APIがbeta→GAへ移行。利用制限が緩和された | false | true | フェーズ2でManaged Agentsを採用候補に昇格させるか再評価が必要 |
```

---

### 記録のタイミングと方法

1. **作業セッション開始時**: 最優先情報源（A-003/A-004/A-001）を確認し、前回確認日以降の変更をスキャンする
2. **変化検知時**: 上記テーブルに1行を即座に追記する。small updateは notes に簡記して followup_required_flag = false とする
3. **重要変更時**: breaking_change_flag または followup_required_flag が true の場合は、対応するitem_idの `last_checked_at` と `notes` を同時に更新する
4. **候補状態変更時**: current_status を変更した場合、update_log に変更理由を記録する（item_id + update_summaryに状態変更の旨を明記）

## パート3｜重要変更の判定基準

### 追跡対象イベント（必須）

- 新機能追加
- 仕様変更
- GA / beta 状態変更
- breaking change
- 主要OSSの大きな更新
- 主要OSSの停滞 / 非推奨化

### 重要度の判定基準

| 重要度 | 判定基準 | 対応 |
|--------|---------|------|
| 緊急（即対応） | breaking changeが確認されたとき / betaからGAへの移行 / 廃止予告（deprecation notice）が出たとき | 当日中に candidate_catalog の対象item更新 + followup_required_flag = true |
| 高（早期対応） | 本プロジェクトで採用済みまたは採用候補に昇格した候補のAPIや設定形式が変更されたとき | 1週間以内に内容確認・item更新 |
| 中（定期対応） | 監視継続・比較待ちの候補に大型機能追加があったとき / 主要OSSのメジャーバージョン更新 | 次回定期確認時にitem更新 |
| 低（記録のみ） | マイナーバグ修正・ドキュメント整備・スター数の増減 | update_logに追記のみ |

## パート4｜再調査の発生条件

### 自動的に再調査が必要になる条件

以下のいずれかが発生した場合、対象候補を「調査中」状態に戻し、再調査を実施する。

1. **breaking change検知**: 採用候補・試験導入候補において、実装に影響するbreaking changeが確認された場合
2. **beta状態変化**: betaであることを理由に「保留」としている候補がGA移行または廃止されたとき
3. **競合の不採用決定**: 比較対象の一方が不採用候補になったとき（残った候補の詳細調査が必要になる）
4. **新たな一次情報の出現**: 既存候補に関するAnthropicの公式発表（ブログ・リリースノート）が出たとき
5. **6ヶ月以上の未確認**: last_checked_atが6ヶ月以上前の候補は自動的に再調査対象とする

### 手動で再調査を入れるべき条件

1. **実装着手直前**: フェーズ2以降で候補の実装着手前に必ず最新状態を確認する
2. **比較評価着手前**: 複数候補を比較評価する前に全候補を最新状態に更新する
3. **異常な活動低迷**: GitHubリポジトリのコミットが3ヶ月以上停止し、Issueにも反応がない場合
4. **ユーザーからの指示**: プロジェクト担当者から特定候補の再確認指示があった場合

## パート5｜既存候補の再確認条件

一度登録した候補をいつ再確認するかの条件。

| 再確認トリガー | 対象候補の範囲 | 優先度 |
|-------------|-------------|--------|
| 最優先情報源（A-003/A-004）に当該候補の記載が出た | 当該候補 | 緊急 |
| 作業セッション開始時のスキャン（毎回） | 採用候補・試験導入候補 | 高 |
| 月次定期スキャン（月1回） | 調査中・候補・比較待ち・監視継続 | 中 |
| 6ヶ月経過 | last_checked_atが6ヶ月以上前の全候補 | 中 |
| フェーズ移行時（フェーズ2着手前など） | 全候補 | 高 |
| 候補状態変更の検討時 | 変更対象候補 + related_itemsに列挙された候補 | 高 |
| 保留理由の解消条件が成立したとき | 保留状態の当該候補 | 高 |

## パート6｜Claude Code 自律調査ルール

Claude Code が自分から追加調査を行うための絶対ルール。このルールはプロジェクト全期間を通じて遵守する。

### 既知候補依存の禁止

**ルール**: `02_candidate_catalog.md` に登録済みの候補だけを前提として調査・提案を完結させてはならない。

**適用場面**: 新しい機能領域や比較評価を行うとき、まず「カタログ外の選択肢が存在しないか」を確認する調査を先に行う。

**具体的行動**: 調査開始時に必ず「カタログ未登録だが存在する可能性がある候補」の探索を行い、見つかった場合はカタログに追加してから比較評価に進む。

---

### 一次情報優先ルール

**ルール**: 情報の根拠として使用するURLは、必ず `01_information_sources.md` に定義されたカテゴリAの一次情報源、またはそれと同等の公式情報源を優先する。

**禁止事項**: ブログ記事・Qiita・Reddit等の二次情報のみを根拠として採用判断を行ってはならない。

**適用例外**: 二次情報は「調査のヒント」として使用することは許容する。ただし必ずその後に一次情報での確認を行い、確認した一次情報URLをevidence_linksに記録する。

---

### GitHub再探索の義務

**ルール**: ある機能領域（例: evaluation/observability）について調査するとき、既存カタログを参照するだけでなく、GitHubの関連キーワード検索を必ず実施する。

**義務化の理由**: カタログは過去時点のスナップショットに過ぎず、急成長している新興OSSを見落とすリスクがある。

**実行頻度**: フェーズ2移行時・各領域の比較評価着手時・6ヶ月毎の定期スキャン時に実施する。

---

### 「これで全部」禁止ルール

**ルール**: 調査結果を提示する際に「これが全ての選択肢です」という断言をしてはならない。

**義務**: 調査の限界・確認できなかった範囲を常に明示する。「今回の調査範囲では〜」「以下は把握できた候補だが、他にも存在する可能性がある」という表現を使う。

**具体的記録**: `05_unexplored_and_phase2_handoff.md` の「未探索領域」セクションを常に最新に保ち、不確実な領域を明示的に管理する。

---

### 比較候補の複数提示ルール

**ルール**: 採用候補の提案・比較評価を行うとき、必ず2つ以上の選択肢を並べて提示する。

**例外なし**: たとえ1つの選択肢が明らかに優位であると判断される場合でも、代替候補を挙げた上でなぜその選択肢が優位かを説明する。

**記録義務**: 比較評価の結果は `alternative_candidates` フィールドおよびフェーズ2の比較評価ドキュメントに記録する。

---

### 保留・未確認の積極的解消ルール

**ルール**: 現在「保留」または「未確認」状態の候補（特にカテゴリD・E）は、次フェーズ開始時までに状態を前進させる努力をする。

**具体的行動**: 各フェーズ開始前に `02_candidate_catalog.md` の「未確認」「保留」候補を一覧し、解消できる候補から調査を着手する。

**例外**: `05_unexplored_and_phase2_handoff.md` の「保留候補と保留理由」に明示的に「フェーズN以降まで保留」と記録された候補は、そのフェーズに達するまで調査不要とする。
