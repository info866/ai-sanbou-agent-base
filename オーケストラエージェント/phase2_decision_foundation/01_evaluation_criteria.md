# 評価基準一覧（Evaluation Criteria）

> **担当**: グループ1  
> **ステップ対応**: フェーズ2作業指示書 ステップ2（評価基準の固定）  
> **最終更新**: 2026-04-12（グループ1）

---

## この文書の役割

本文書は、フェーズ2で全候補を評価する際の**共通物差し**を定義する。後続グループ（特にグループ3: 候補別評価記録作成）は、この10軸を使って全候補を評価すること。

---

## 評価の基本原則

1. **全候補に同じ物差しを適用する**: 公式機能もOSSも同じ10軸で評価する
2. **曖昧語だけで終えない**: 「良い」「悪い」ではなく、具体的な条件で判定する
3. **star数や話題性だけで代理評価しない**: 人気は参考情報に留め、目的適合性で判断する
4. **根拠を明示する**: 評価の根拠となった情報源（URL・ドキュメント・リポジトリ）を必ず記録する
5. **不明点は「不明」と記録する**: 無理に評価せず、不明点は未解決論点として残す

---

## 評価スコアの付け方

各評価軸に対して、以下の5段階で評価する。

| スコア | 意味 |
|--------|------|
| 5 | 非常に高い（本プロジェクトにとって理想的） |
| 4 | 高い（十分に適合する） |
| 3 | 中程度（条件付きで適合する） |
| 2 | 低い（懸念がある） |
| 1 | 非常に低い（ほぼ不適合） |
| — | 評価不能（情報不足で判定できない。未解決論点に記録すること） |

---

## 根拠に使う情報の優先順位

| 優先度 | 情報種別 | 例 |
|--------|---------|-----|
| 1（最優先） | 公式一次情報 | Anthropic公式docs、リリースノート、公式GitHub README |
| 2 | 公式実装・公式サンプル | claude-agent-sdk-demos、公式action、公式registry |
| 3 | OSSリポジトリの直接情報 | GitHub README、CHANGELOG、Issues、コミット履歴 |
| 4 | フェーズ1成果物の記録 | 02_candidate_catalog.md の記録内容 |
| 5（最低） | 二次情報 | ブログ記事・Qiita・Reddit等（ヒントとしてのみ使用可） |

**禁止**: 二次情報のみを根拠として採用判断を行ってはならない。必ず一次情報で確認すること。

## 評価軸の定義

### E1｜目的一致度（Purpose Fit）

| 項目 | 内容 |
|------|------|
| **何を見る軸か** | この候補が、最終ゴールであるAI参謀エージェント基盤の構築にどれだけ直接効くか。今のフェーズおよび今後のフェーズにどれだけ関係が深いか |
| **高評価の条件** | AI参謀基盤の中核機能（情報収集・評価・採用判断・実行投入・能力最適化提案）に直接寄与する。複数フェーズにまたがって利用価値がある |
| **低評価の条件** | 本プロジェクトの最終目的と接点が薄い。汎用的すぎて参謀基盤への具体的貢献が説明できない。他のプロジェクトでは有用だが今回のゴールには遠い |
| **根拠に使う情報** | 大分類要件提起書のゴールG1〜G6、フェーズ2要件定義書の対象範囲、候補のprimary_use_cases |
| **後続グループでの使い方** | 目的一致度が3以下の候補は、保留または監視継続への分類を検討する根拠となる |

---

### E2｜役割明確性（Role Clarity）

| 項目 | 内容 |
|------|------|
| **何を見る軸か** | この候補が何のために使うものかが明確か。他の候補と役割重複が大きすぎないか |
| **高評価の条件** | 「何に使うか」が1〜2文で明確に説明できる。本プロジェクトのアーキテクチャ上の位置（L01〜L08のどこに入るか）が明確。他候補との役割境界がはっきりしている |
| **低評価の条件** | 何に使うか説明に迷う。複数の役割を兼ねすぎていて位置づけが曖昧。他候補との重複が大きく、どちらを使うべきか判断しにくい |
| **根拠に使う情報** | 公式ドキュメントの概要説明、03_taxonomy_and_state.md の機能層定義（L01〜L08）、候補間のrelated_items |
| **後続グループでの使い方** | 役割が重複する候補同士の比較評価の起点になる。重複が大きい場合、どちらかを不採用にする判断根拠となる |

---

### E3｜導入価値（Adoption Value）

| 項目 | 内容 |
|------|------|
| **何を見る軸か** | この候補を導入すると、時間短縮・品質向上・抜け漏れ削減に効くか。ユーザーのボトルネック解消に寄与するか |
| **高評価の条件** | 導入することで明確な時間短縮・品質向上・自動化が実現する。手作業や勘に頼っていた部分を構造的に解決する。導入しない場合のリスク（見落とし・品質低下等）が大きい |
| **低評価の条件** | 導入しても既存のやり方と大差ない。改善が限定的で導入コストに見合わない。すでに他の手段で同等の効果が得られている |
| **根拠に使う情報** | 候補のprimary_use_cases、公式ドキュメントのユースケース記述、既存ワークフローとの差分分析 |
| **後続グループでの使い方** | 導入価値が4以上の候補は、採用候補または試験導入候補への昇格を検討する根拠となる |

---

### E4｜流用可能性（Reusability）

| 項目 | 内容 |
|------|------|
| **何を見る軸か** | 既存資産をそのまま流用できるか、一部流用できるか、設計思想だけ参考になるか、流用価値が低いか |
| **高評価の条件** | そのまま本プロジェクトに取り込んで機能する（全面流用可能）。カスタマイズ箇所が少ない。APIやインターフェースが安定している |
| **低評価の条件** | 本プロジェクトの要件に合わせるには大幅な改変が必要。設計思想は参考になるが実装はゼロから書く方が早い。依存関係が重すぎて部分流用も困難 |
| **根拠に使う情報** | OSSのREADME・設定方法・カスタマイズ例、公式機能のAPI仕様・設定形式、03_taxonomy_and_state.md のreuse_possibility定義 |
| **後続グループでの使い方** | 自作/流用判断（B1〜B5）の直接的な判定根拠。E4が4以上なら全面流用（B1）または部分流用（B2）を検討。E4が2以下なら自作優先（B4）を検討 |

---

### E5｜導入難易度（Implementation Burden）

| 項目 | 内容 |
|------|------|
| **何を見る軸か** | セットアップが重すぎないか。維持コストが高すぎないか。今のプロジェクト段階に対して重すぎないか |
| **高評価の条件**（=導入が容易） | セットアップが10分〜1時間程度で完了する。依存関係が少ない。ドキュメントが充実していて迷わない。既存環境との競合がない |
| **低評価の条件**（=導入が困難） | セットアップに複雑なインフラ構築が必要。依存関係が多く競合リスクがある。ドキュメントが不十分で試行錯誤が必要。維持に継続的なコストがかかる |
| **根拠に使う情報** | 公式のインストール手順・prerequisites、OSSのREADME・Getting Started、GitHubのissue傾向（セットアップ困難の報告） |
| **後続グループでの使い方** | E5が2以下の候補は、試験導入候補として慎重に扱う。E5が4以上の候補は、早期試験導入の候補として優先する |

**注意**: E5は「難しいほど低スコア」とする。導入が容易な方が高スコア。

---

### E6｜保守性 / 継続性（Maintainability）

| 項目 | 内容 |
|------|------|
| **何を見る軸か** | 更新が継続しているか。保守されているか。将来すぐ死蔵しそうでないか |
| **高評価の条件** | 直近3ヶ月以内にアクティブな更新がある。コントリビューターが複数いる。公式（Anthropic等）が主導している。ロードマップまたは継続的なリリースが確認できる |
| **低評価の条件** | 最終更新が6ヶ月以上前。メンテナーが1人で活動が不安定。issueが放置されている。deprecated表示やarchived状態 |
| **根拠に使う情報** | GitHubのlast commit日・コントリビューター数・issue対応状況、公式ドキュメントの更新日、04_update_tracking_rules.md のmaintenance_status定義 |
| **後続グループでの使い方** | E6が2以下の候補は、監視継続または保留への分類を検討する根拠となる。採用候補に昇格させる場合は保守リスクを明記する |

---

### E7｜安全性 / リスク（Safety & Risk）

| 項目 | 内容 |
|------|------|
| **何を見る軸か** | 信頼境界の問題がないか。権限や外部接続のリスクが高すぎないか。ブラックボックス依存が大きすぎないか |
| **高評価の条件** | 権限モデルが明確（最小権限原則）。外部接続先が限定的で制御可能。コードが公開されていて動作が検証可能。APIキー等の機密情報の扱いが適切 |
| **低評価の条件** | 広範な権限を要求する。外部サービスへの無制限アクセスを前提とする。ブラックボックス部分が大きく検証困難。機密情報の取り扱いが不透明 |
| **根拠に使う情報** | 公式ドキュメントの権限・セキュリティセクション、OSSのライセンス・コードレビュー、大分類要件提起書の要件R8（安全性） |
| **後続グループでの使い方** | E7が2以下の候補は、リスク軽減策を記録した上でのみ採用候補とする。E7が1の場合は不採用候補への分類を検討 |

---

### E8｜拡張性（Extensibility）

| 項目 | 内容 |
|------|------|
| **何を見る軸か** | 今後のフェーズ3〜5に接続しやすいか。将来の追加機能や追加MCPを取り込みやすいか |
| **高評価の条件** | プラグイン機構やAPI拡張ポイントが用意されている。MCP対応やフック機構で後から機能追加できる。フェーズ3（知識活用）・フェーズ4（実行基盤）・フェーズ5（AI参謀完成）の要件に接続しやすい |
| **低評価の条件** | 閉じた設計で拡張が困難。特定ユースケースに特化していて汎用的な拡張ができない。フェーズ3〜5の要件と接続する具体的な方法が見えない |
| **根拠に使う情報** | 公式ドキュメントの拡張機能説明、OSSのプラグインアーキテクチャ、大分類要件提起書のフェーズ3〜5定義 |
| **後続グループでの使い方** | E8が4以上の候補は、フェーズ3引き継ぎ候補として優先的に記録する。E8が2以下の候補は、現フェーズ限りの利用に留めることを検討する |

---

### E9｜代替可能性（Alternatives Availability）

| 項目 | 内容 |
|------|------|
| **何を見る軸か** | 同じ役割の代替候補があるか。より軽い候補、より安定した候補、より公式に近い候補があるか |
| **高評価の条件**（=代替が少ない=この候補の独自価値が高い） | 同じ役割を果たせる代替候補がほとんどない。唯一無二の機能や公式サポートがある。代替に切り替えるとコストが大幅に増加する |
| **低評価の条件**（=代替が多い=この候補でなくてもよい） | 同じ役割を果たす代替候補が複数存在する。他の候補の方がより適合する可能性がある。容易に差し替えられる |
| **根拠に使う情報** | 02_candidate_catalog.md のalternative_candidates、比較グループ（02_comparison_units_and_matrix_plan.md）の構成 |
| **後続グループでの使い方** | E9が高い（代替が少ない）候補は、優先的に評価・採用検討する。E9が低い（代替が多い）候補は、比較評価を通じて最適な1つを選定する |

**注意**: E9は「代替が少ないほど高スコア」とする。代替が多いことは候補の弱みではないが、比較評価の必要性が高いことを示す。

---

### E10｜今やる意味（Timeliness）

| 項目 | 内容 |
|------|------|
| **何を見る軸か** | 今この段階で触る価値があるか。今は保留にして後でよいか |
| **高評価の条件** | 今触らないと後続フェーズがブロックされる。GA済みで安定しており今すぐ活用可能。フェーズ2の作業品質を直接向上させる。学習コストの前倒しに明確なメリットがある |
| **低評価の条件** | beta/experimentalで安定性に不安がある。フェーズ4〜5で初めて必要になる。今触っても後で仕様変更で手戻りするリスクがある。他の候補の評価が先に必要 |
| **根拠に使う情報** | 候補のbeta_or_ga状態、フェーズ要件定義書の対象範囲（今回やること/やらないこと）、05_unexplored_and_phase2_handoff.md の優先論点 |
| **後続グループでの使い方** | E10が4以上の候補は、最優先（P1）または高優先（P2）への分類を検討する。E10が2以下の候補は、低優先（P4）または凍結（P5）を検討する |

## 評価結果の記録フォーマット

グループ3が `04_candidate_evaluations.md` に記録する際の1候補あたりのフォーマット:

```
### F-XXX | 候補名

| 評価軸 | スコア | 根拠（1〜2文） |
|--------|--------|--------------|
| E1 目的一致度 | X | ... |
| E2 役割明確性 | X | ... |
| E3 導入価値 | X | ... |
| E4 流用可能性 | X | ... |
| E5 導入難易度 | X | ... |
| E6 保守性 | X | ... |
| E7 安全性 | X | ... |
| E8 拡張性 | X | ... |
| E9 代替可能性 | X | ... |
| E10 今やる意味 | X | ... |

- **compared_with**: 比較対象のitem_id
- **decision_state**: 判断出力（D1〜D5のいずれか）→ グループ4が確定
- **build_vs_buy_state**: 自作/流用判断（B1〜B5のいずれか）→ グループ4が確定
- **current_priority**: 優先度（P1〜P5のいずれか）→ グループ4が確定
- **reason_summary**: 判断理由の要約
- **unresolved_points**: 未解決の論点
- **next_review_condition**: 再確認をトリガーする条件
- **next_action**: 次に取るべきアクション
```

## 判断出力の定義（参照用）

グループ4が使う判断分類。ここでは参照用に転記する。

| ID | 判断 | 意味 |
|----|------|------|
| D1 | 採用候補 | 今後の基盤に組み込む方向で優先的に検討・試験すべき |
| D2 | 試験導入候補 | 本採用前に小さく試す価値が高い |
| D3 | 保留 | 価値はありそうだが時期尚早・情報不足・比較不足で決めない |
| D4 | 不採用 | 現時点では使わない（理由と再検討条件を記録） |
| D5 | 監視継続 | 今は使わないが、更新や成熟度の変化を追う |

---

## 自作/流用判断の定義（参照用）

| ID | 判断 | 意味 |
|----|------|------|
| B1 | 全面流用 | 既存資産をそのまま基盤として採用する価値が高い |
| B2 | 部分流用 | 一部機能・一部設計のみ流用し、自作補完する |
| B3 | 参考流用 | 設計思想や構成だけ参考にし、実装は自作する |
| B4 | 自作優先 | 既存候補では要件に合わず、自作した方がよい |
| B5 | 判断保留 | 追加比較・追加調査が必要でまだ決めない |

---

## 優先順位の定義（参照用）

| ID | 優先度 | 意味 |
|----|--------|------|
| P1 | 最優先 | 次フェーズで必ず深掘り・再利用・試験導入判断に進む |
| P2 | 高優先 | 重要性は高いが、最優先候補の直後に扱う |
| P3 | 中優先 | 価値はあるが、今すぐではなくてもよい |
| P4 | 低優先 | 監視継続で足りる |
| P5 | 凍結 | 現時点では積極的に扱わず、再評価条件が満たされるまで凍結 |
