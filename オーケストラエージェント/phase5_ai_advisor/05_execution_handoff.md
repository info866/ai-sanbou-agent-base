# 実行基盤への橋渡しルール（Execution Handoff Rules）

> **正本**: 11.フェーズ5要件定義書.md  
> **最終更新**: 2026-04-13

---

## この文書の役割

AI参謀の提案（04_reason_generation.md の出力）を、Phase 4実行基盤（phase4_execution_foundation/）へそのまま渡せる形に変換するルールを定義する。

---

## 橋渡しの5要素

AI参謀の提案から実行基盤への橋渡しでは、以下の5要素を確定させる。

### 要素1｜対象

何に対して作業するかを特定する。

- ファイルパス（既存ファイルの修正なら具体的パス）
- 新規作成物の名前と配置先
- 外部リソース（GitHub URL、公式ドキュメントURL等）

### 要素2｜使用能力

能力選定結果（03_capability_selection.md）から確定した能力リストを、実行順序付きで渡す。

```
使用能力（実行順）:
  1. [F-XXX] 能力名 — ステップ: 調査
  2. [F-XXX] 能力名 — ステップ: 実装
  3. [F-XXX] 能力名 — ステップ: 検証
```

### 要素3｜作業順

Phase 4の実行フロー（01_execution_flow.md）に基づく作業順序。

```
作業順:
  1. 調査 → [具体的に何を調査するか]
  2. 実装 → [具体的に何を作るか]
  3. 検証 → [具体的に何をテストするか]
  4. GitHub反映 → [コミットメッセージの方向性]
```

不要なステップは省略する。全7ステップを常に実行する必要はない。

### 要素4｜検証方法

Phase 4の品質保証ルール（05_quality_assurance_rules.md）に基づく検証方法。

**4階層検証の適用:**
1. 構文検証 — コードなら構文チェック、文書なら形式確認
2. インポート検証 — 依存関係の確認
3. 機能検証 — 期待動作の確認
4. パフォーマンス/セキュリティ検証 — 必要時のみ

検証方法は依頼分類に応じて適用レベルを変える:
- RC-3実装中心/RC-4修正中心 → 4階層全て必須
- RC-1調査中心/RC-2比較中心 → 構文検証+機能検証
- RC-5設計中心 → 形式確認+一貫性確認
- RC-6運用改善中心 → 機能検証+動作確認

### 要素5｜GitHub反映条件

Phase 4のGitHub統合方針（06_github_integration_policy.md）に基づく反映条件。

- コミット単位: 1タスク=1コミットが原則
- コミットメッセージ: type(scope): description 形式
- PR要否: 複数コミットにまたがる変更、またはレビューが必要な変更はPR作成
- push先: origin/main（本プロジェクトの標準）

## 橋渡し出力テンプレート

```
実行指示:
  対象: [ファイルパス/リソース]
  
  使用能力:
    1. [F-XXX] — [ステップ名]: [具体的用途]
    2. [F-XXX] — [ステップ名]: [具体的用途]
  
  作業順:
    1. [ステップ] — [具体内容]
    2. [ステップ] — [具体内容]
  
  検証方法:
    - [検証項目1]: [確認方法]
    - [検証項目2]: [確認方法]
  
  GitHub反映:
    - コミット: [type(scope): description]
    - PR: [要/不要]
    - push: origin/main
```

## 橋渡しの品質確認

橋渡し出力を提出する前に、以下を確認する。

1. 対象が具体的か（「何か」ではなく「このファイル」）
2. 使用能力がitem_idで特定されているか
3. 作業順がPhase 4の実行フローと整合しているか
4. 検証方法が具体的か（「テストする」ではなく「何をどう確認するか」）
5. GitHub反映条件が決まっているか

5項目全てYESなら橋渡し完了。1つでもNOなら不足を補完する。
