# 理由生成ルール（Reason Generation & Proposal Output Rules）

> **正本**: 11.フェーズ5要件定義書.md  
> **最終更新**: 2026-04-13

---

## この文書の役割

能力選定結果に対して、選定理由を生成し、提案として出力するルールを定義する。理由のない判断は禁止であり、全ての選定・不選定に根拠を付ける。

---

## 理由生成の3要素

すべての選定理由は、以下の3要素を含む。

### 要素1｜なぜ使うか（正の理由）

選定した能力について:
- 目的のどの部分に直接効くか
- Phase 2の評価結果（D1/D2/P1/P2）のどれに該当するか
- 具体的にどの作業ステップで使うか

**例:** 「F-003 Subagentsを使う。理由: 並列調査が必要な依頼であり、D1採用候補・P1最優先。調査ステップで複数情報源を同時に確認するのに直接効く。」

### 要素2｜なぜ他を使わないか（消去理由）

不選定とした能力について:
- 適合判定（S1〜S8）のどの基準で外れたか
- 再検討条件があれば明示

**例:** 「F-027 LangGraphは使わない。理由: S2不適合（今回の調査にはオーケストレーション不要）。Agent SDK単体での限界に達した場合に再検討。」

### 要素3｜どこに効くか（効果範囲）

選定した能力が、Phase 4実行フローのどのステップに効くかを明示する。

Phase 4の7ステップ（01_execution_flow.md）:
1. 調査
2. 比較
3. 実装
4. 修正
5. 検証
6. 記録
7. GitHub反映

**例:** 「F-004 Hooksは検証ステップとGitHub反映ステップに効く。PreCommitフックで品質検証を自動化し、PostCommitフックで記録を自動化する。」

## 提案出力の必須8項目

AI参謀の提案は、以下の8項目を必ず含む。

### 1. 今回の目的
依頼から特定した目的。1〜2文で記述。

### 2. 今回使うべき能力
item_idと能力名のリスト。各能力の役割を1文で付記。

### 3. 今回使わない能力
item_idと能力名のリスト。不選定理由を1文で付記。
候補プールに入ったが選ばなかった能力のみ記載（候補プール外は省略）。

### 4. 理由
上記3要素（正の理由・消去理由・効果範囲）をまとめた理由文。

### 5. 再確認の要否
再確認が必要な能力・情報のリスト。不要なら「再確認不要」と明記。

### 6. 推奨作業フロー
Phase 4の7ステップのうち、今回実行するステップの順序。不要なステップは省略。

### 7. 完了条件
今回の依頼が完了したとみなす条件。検証可能な形で記述。

### 8. 検証条件
完了を検証する具体的な方法。テスト・差分確認・動作確認等。

## 提案出力テンプレート

```
目的: [1〜2文]

使う能力:
  - [F-XXX] 能力名 — [役割]
  - [F-XXX] 能力名 — [役割]

使わない能力:
  - [F-XXX] 能力名 — [不選定理由]

理由:
  [正の理由・消去理由・効果範囲を統合した記述]

再確認: [要/不要。要の場合は対象と方法]

作業フロー:
  1. [ステップ] — [使う能力]
  2. [ステップ] — [使う能力]

完了条件:
  - [条件1]
  - [条件2]

検証条件:
  - [方法1]
  - [方法2]
```

## 理由の品質基準

以下を満たさない理由は不十分であり、書き直す。

| 基準 | 判定方法 |
|------|---------|
| 具体性 | item_idまたは能力名が含まれているか |
| 根拠性 | Phase 2の評価結果・Phase 3の知識を参照しているか |
| 反証性 | 不選定理由が「なんとなく不要」ではなくS1〜S8の基準で説明されているか |
| 接続性 | Phase 4の実行ステップと紐付いているか |
