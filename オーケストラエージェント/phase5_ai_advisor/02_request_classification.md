# 依頼分類ルール（Request Classification Rules）

> **正本**: 11.フェーズ5要件定義書.md  
> **最終更新**: 2026-04-13

---

## この文書の役割

ユーザーの依頼を受け取ったとき、最初に実行する分類判定のルールを定義する。分類結果が後続の能力選定・理由生成・実行橋渡しの全てを方向付ける。

---

## 分類体系

依頼は以下の6類型のいずれかに分類する。複合依頼は主軸を1つ選び、副軸を注記する。

### RC-1｜調査中心

**判定条件:** 「調べてほしい」「何があるか知りたい」「現状を確認したい」
- 情報収集が主目的
- 比較・実装は含まない、または従属的
- 出力は情報の整理・要約

**対応する知識活用フロー:** F1→F2→F3→F4→F5→F6（phase3_knowledge_foundation/04_knowledge_utilization_flow.md）

**対応する作業単位:** 調査単位（phase4_execution_foundation/04_work_unit_definitions.md）

---

### RC-2｜比較中心

**判定条件:** 「どちらがいいか」「比較してほしい」「選んでほしい」
- 複数候補の優劣判定が主目的
- 採用/保留/不採用の決定を含む

**対応する知識活用フロー:** SR-04（比較結果確認）を主軸とする

**対応する作業単位:** 比較単位（phase4_execution_foundation/04_work_unit_definitions.md）

**参照すべき既存資産:**
- phase2_decision_foundation/01_evaluation_criteria.md（評価基準E1〜E10）
- phase2_decision_foundation/04_candidate_evaluations.md（既存評価結果）

---

### RC-3｜実装中心

**判定条件:** 「作ってほしい」「実装してほしい」「動くものがほしい」
- コード・設定・構成の作成が主目的
- 動作する成果物の納品を含む

**対応する作業単位:** 実装単位（phase4_execution_foundation/04_work_unit_definitions.md）

**必須後続処理:**
- 品質検証（phase4_execution_foundation/05_quality_assurance_rules.md）
- GitHub反映（phase4_execution_foundation/06_github_integration_policy.md）

---

### RC-4｜修正中心

**判定条件:** 「直してほしい」「バグがある」「動かない」「改善してほしい」
- 既存の問題の解消が主目的
- 問題の特定→修正→再検証を含む

**対応する作業単位:** 修正単位（phase4_execution_foundation/04_work_unit_definitions.md）

**必須後続処理:**
- 影響範囲確認（phase4_execution_foundation/05_quality_assurance_rules.md）
- 差分確認（git diff）

---

### RC-5｜設計中心

**判定条件:** 「どう作るべきか」「設計を考えてほしい」「アーキテクチャを決めたい」
- 構造・方針・設計の決定が主目的
- 実装は含まない、または後続タスクとして分離する

**対応する能力:** 主にP1最優先の構造化能力
- F-002 MCP（接続設計）
- F-003 Subagents（分割設計）
- F-009 Claude Agent SDK（エージェント設計）

---

### RC-6｜運用改善中心

**判定条件:** 「効率を上げたい」「自動化したい」「ワークフローを改善したい」
- 既存プロセスの改善が主目的
- 新規ツール導入・既存ツール活用の最適化を含む

**対応する能力:** 主にP1/P2の自動化能力
- F-004 Hooks（自動化トリガー — コミット、イベント駆動）
- F-005 Skills（作業パッケージ化・再利用）
- F-010/F-014 GitHub Actions（CI/CD・PRレビュー自動化）
- F-011 Cloud Scheduled Tasks（/scheduleコマンドによる定期タスク自動実行、D2試験導入候補、P3中優先）

## 分類判定フロー

```
依頼を受け取る
  │
  ├─ 問題・エラーの解消が主目的か?
  │    └─ YES → RC-4（修正中心）
  │
  ├─ 動くコード・設定の作成が主目的か?
  │    └─ YES → RC-3（実装中心）
  │
  ├─ 複数候補の比較・選択が主目的か?
  │    └─ YES → RC-2（比較中心）
  │
  ├─ 構造・方針・アーキテクチャの決定が主目的か?
  │    └─ YES → RC-5（設計中心）
  │
  ├─ 既存プロセスの効率化・自動化が主目的か?
  │    └─ YES → RC-6（運用改善中心）
  │
  └─ 情報の収集・確認・整理が主目的か?
       └─ YES → RC-1（調査中心）
```

**複合依頼の処理:**
- 主軸を1つ選ぶ（最も重要な目的）
- 副軸を注記する（「主: RC-3実装、副: RC-2比較」）
- 作業順は主軸から開始し、副軸を後続で処理

---

## 分類結果の出力形式

```
分類: RC-X（類型名）
副軸: RC-Y（あれば）
判定根拠: 依頼のどの部分がこの分類を示しているか
```
