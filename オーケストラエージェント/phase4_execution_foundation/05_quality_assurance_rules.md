# 品質保証ルール（Quality Assurance Rules）

> **ステップ対応**: フェーズ4作業指示書 ブロック5  
> **最終更新**: 2026-04-13

---

## この文書の役割

自己検証・差分確認・完了報告・影響範囲確認・未完了報告を必須化し、フェーズ4作業の品質を確保する。

---

## ルール1：自己検証は必須

### 原則

**すべての実装作業に自己検証を含める。自己検証なしで「完了」扱いしない。**

### 自己検証の4階層

#### 階層1：構文検証（5分）

```bash
# Python
python -m py_compile [file.py]

# JavaScript/TypeScript
node -c [file.js] OR eslint [file]

# YAML
yamllint [file.yaml]

# JSON
jq . < [file.json]
```

#### 階層2：インポート・依存関係検証（5分）

```bash
# Python
python -c "import [module]; print([module].__version__)"

# Node/JavaScript
npm ls

# Go
go mod verify

# 外部API
curl -H "Authorization: Bearer ${TOKEN}" https://api.../test
```

#### 階層3：機能検証（10〜30分）

```
- Unit test が成功するか
- Manual test で想定動作するか
- エラーハンドリングが機能するか
- Edge case に対応しているか
```

#### 階層4：パフォーマンス・セキュリティ検証（5〜15分）

```
- 処理時間が許容値以内か（T < X秒）
- 認証情報が平文に含まれていないか
- SQL injection / XSS に対応しているか
- 依存ライブラリの脆弱性がないか（pip audit, npm audit）
```

### 自己検証記録フォーマット

各実装完了後、以下を記録：

```
自己検証結果:
  [ ] 階層1（構文）: PASS/FAIL
  [ ] 階層2（インポート）: PASS/FAIL
  [ ] 階層3（機能）: PASS/FAIL
  [ ] 階層4（パフォーマンス）: PASS/FAIL

FAIL が1つでもある場合:
  → STEP 4（修正）へ進む
  → 修正後に全階層を再検証
```

## ルール2：差分確認は必須

### 原則

**すべての変更を git diff で確認し、意図通りの変更であることを証明する。**

### 差分確認フロー

```
$ git status
  M [ファイルA]
  A [ファイルB]
  D [ファイルC]

各ファイルの差分を確認:
$ git diff [ファイルA]

確認項目:
  ✓ 意図しない変更（削除・空白・コメント）がないか
  ✓ 必要な変更がすべて含まれているか
  ✓ コード品質は許容か（インデント・スタイル）
```

### 差分確認記録フォーマット

```
変更サマリー:
  追加ファイル: [ファイル一覧]
  修正ファイル: [ファイル一覧]
  削除ファイル: [ファイル一覧]

各ファイルの確認結果:
  [ファイル]: 意図通り ✓ / 修正必要 ✗
```

## ルール3：完了条件チェックリスト

### 原則

**すべての作業に「完了条件」を定義し、チェックリストで検証。**

### 完了条件の例

```
## 実装タスク：LiteLLM 統合

完了条件:
  [ ] pip install litellm が成功する
  [ ] import litellm がエラーなし
  [ ] テストコールで複数モデル（Claude 4.6, 4, 3.5 Sonnet）に接続できる
  [ ] 応答時間が 5秒以内
  [ ] API key が .env に隔離されている
  [ ] git diff で差分が確認できる

→ すべてチェック後に「完了」
```

## ルール4：影響範囲確認

### 原則

**今回の変更が他のファイル・機能に悪影響を与えないことを確認。**

### 影響範囲確認フロー

```
変更対象ファイル: [ファイルA]
  ↓
関連ファイルを洗い出す（import, 呼び出し, データ形式依存）
  ├─ [ファイルB]: import [ファイルA] → B の動作確認
  ├─ [ファイルC]: [ファイルA].func() 呼び出し → C の機能テスト
  └─ [ファイルD]: [ファイルA] のデータ形式に依存 → D の形式テスト
  ↓
各ファイルで影響テスト実施
  ↓
副作用なし → 確認完了
副作用あり → 設計レベルの問題か実装上の問題か判定 → 修正
```

### 影響範囲確認記録

```
変更ファイル: [ファイルA]
関連ファイル: [ファイル一覧]
各ファイルへの影響テスト:
  [ファイルB]: [内容] → PASS/FAIL
  [ファイルC]: [内容] → PASS/FAIL
副作用: なし / あり（[内容]）
```

## ルール5：未完了報告

### 原則

**スコープ外・後回し・問題のある事項を隠さず、明示的に記録する。**

### 未完了報告フォーマット

```
## 未完了事項

### スコープ外（意図的に今回はやらない）
  - [項目]: 理由「[理由]」→ [いつ対応か]

### 後回し（計画内だが今回は実施できなかった）
  - [項目]: 理由「[理由]」→ [いつ対応か]

### 課題（実施したが、問題が残っている）
  - [項目]: 問題「[内容]」→ 対応「[対応内容]」

### 意思決定待ち（判断が必要だが、判定保留）
  - [項目]: 待ち理由「[理由]」→ [誰が、いつ決定するか]
```

## ルール6：エラー・問題の隠蔽禁止

### 禁止パターン

❌ 「テストで FAIL が出たが、忙しいから無視する」
❌ 「副作用が見つかったが、後で修正しよう」
❌ 「import エラーがあるが、今は進める」

### 対応方法

**問題が見つかったら、その場で以下のいずれかを実施：**

1. **その場で修正**: 問題を解決し、再検証を実施（STEP 4）
2. **一時的に隔離**: 問題箇所をコメント化し、「未完了」として報告
3. **ロールバック**: 前回の良好状態に戻し、別アプローチを検討

**決して「問題を隠して進める」はしない。**

## ルール7：品質基準

### パフォーマンス基準

| 項目 | 基準 |
|-----|-----|
| API 呼び出し応答時間 | < 5秒（ローカル）/ < 10秒（クラウド） |
| ファイル処理時間 | < 1秒 / 1000行 |
| メモリ使用量 | < 1GB / 実行プロセス |
| テスト実行時間 | < 30秒 / テストスイート |

### セキュリティ基準

| 項目 | 基準 |
|-----|-----|
| API key 平文格納 | 禁止（`.env` に隔離） |
| SQL 注入対策 | パラメータ化クエリ必須 |
| XSS 対策 | ユーザー入力のサニタイズ必須 |
| 脆弱性スキャン | npm audit / pip audit で警告ゼロ |

### コード品質基準

| 項目 | 基準 |
|-----|-----|
| インデント統一 | スペース 2 or 4（プロジェクト内で統一） |
| 行長 | < 120文字 |
| コメント | 複雑ロジックのみ（自明なコード自体がドキュメント） |
| 命名規則 | snake_case（Python）/ camelCase（JS）/ PascalCase（ClassName） |

## ルール8：品質チェックリスト（全作業完了時）

### 全作業完了時の最終チェック

```
## フェーズ4 品質保証最終確認

### 自己検証
  [ ] すべての実装で階層1〜4の検証を実施した
  [ ] FAIL はすべて修正され、再検証で PASS した
  [ ] 検証結果がすべて記録されている

### 差分確認
  [ ] すべての変更ファイルを git diff で確認した
  [ ] 意図しない変更がない
  [ ] 必要な変更がすべて含まれている

### 完了条件確認
  [ ] すべての作業に完了条件が定義されている
  [ ] すべての完了条件がチェックされている
  [ ] 未チェック項目がない

### 影響範囲確認
  [ ] すべての変更に対して関連ファイルを洗い出した
  [ ] 各関連ファイルで影響テストを実施した
  [ ] 副作用がないことを確認した

### 未完了報告
  [ ] スコープ外事項が明記されている
  [ ] 後回し事項が明記されている（対応時期付き）
  [ ] 課題が明記されている（対応内容付き）
  [ ] 隠蔽事項がない

### コード品質
  [ ] パフォーマンス基準を満たしている
  [ ] セキュリティ基準を満たしている
  [ ] コード品質基準を満たしている

→ すべてチェック後に「品質保証完了」
```
