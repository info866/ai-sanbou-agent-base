# 【フェーズ1要件定義書 / タスク名】
お前の今回の任務は、**AI参謀エージェント基盤の最初の土台として、情報基盤フェーズを定義し、今後の判断・検索・採用・実行投入の前提となる情報収集・整理・状態管理の基盤要件を固定すること**である。

この文書は、プロジェクト全体のうち **フェーズ1（情報基盤フェーズ）** の要件定義書である。  
ここで定義するのは、**何を集めるか、どこから集めるか、どう保存するか、どう分類するか、どう更新追跡するか、どう次フェーズへ接続するか** である。  
この文書の段階で、完成版AI参謀・最終オーケストレーター・全自動実行基盤を作ろうとするな。  
このフェーズは、**最終ゴールへ到達するための最初の土台づくり** である。

---

# 【このフェーズの位置づけ】
このフェーズは、全体計画の中で最初に位置する。  
全体の大分類フェーズは以下である。

1. **フェーズ1：情報基盤**
2. **フェーズ2：判断基盤**
3. **フェーズ3：知識活用基盤**
4. **フェーズ4：実行基盤**
5. **フェーズ5：AI参謀完成**

今回扱うのは、このうち **フェーズ1のみ** である。  
フェーズ1の役割は、今後すべての判断と実務投入の前提となる **情報の正規収集・保存・整理・追跡・再利用の基盤** を作ることにある。

---

# 【最重要目的】
このフェーズの最重要目的は以下である。

1. **追跡対象となる公開情報源を固定すること**
2. **収集対象をカテゴリ別に構造化すること**
3. **新機能・MCP・API・OSS・既存エージェントを継続管理できる状態を作ること**
4. **後続フェーズで評価・採用判断・検索・実行投入に使える粒度で情報を保持すること**
5. **Claude Code が、ユーザーの思いつきだけに依存せず、自律的に追加調査を継続する前提を組み込むこと**

---

# 【背景】
このフェーズで解決すべき問題は、単なる「情報不足」ではない。  
本質は、**Anthropic や OpenAI の機能追加、MCP ecosystem の拡張、GitHub上のOSSや既存エージェント実装の増加速度に、人間単独では追従しきれないこと** である。

特に Anthropic 側では、Claude Code 本体だけでなく、Prompt Improver、MCP、subagents、hooks、skills、Claude Managed Agents、Cowork など、**能力が複数レイヤーに分かれて継続追加されている**。  
そのため、フェーズ1では単に記事やURLを集めるのではなく、**何が存在し、どの層に属し、何に効き、どのような作業で使う候補になり、どの既存OSSや運用パターンと接続しうるか** を整理できる状態にしなければならない。

また、既に世の中には優れたOSS、MCPネイティブな agent framework、router、evaluation 基盤、既存エージェントが存在している可能性が高い。  
それらを見落としてゼロから再発明することは無駄である。  
したがって、このフェーズは **「何があるかを継続的に把握し、次フェーズで比較評価できる状態にする」こと** を目的とする。

# 【Claude Code の自律調査要件】
このフェーズでは、Claude Code の自律性を必ず担保しろ。  
以下は必須要件である。

## 要件A1｜既知候補依存禁止
ユーザーが挙げた機能名、こちらが事前に挙げた候補、過去に調べた候補だけで探索を終えるな。

## 要件A2｜一次情報優先
まず公式一次情報を見ろ。  
機能の有無、正式名称、提供状態、関連ドキュメント、推奨ワークフロー、注意点は、必ず一次情報を正本にしろ。

## 要件A3｜GitHub再探索必須
事前に見つかったリポジトリがある場合でも、それだけで十分だと思うな。  
毎回、現在の目的に照らして **新たに GitHub や公開OSS を探索し直す** 前提で動け。

## 要件A4｜「これで全部」禁止
調査結果を出すとき、「これが全て」と断定するな。  
必ず、
- 今回見つかった有力候補
- 今回未確認の領域
- 今後追加探索すべき領域
を分けて扱え。

## 要件A5｜比較候補の複数提示
1つの候補だけ見つけて満足するな。  
少なくとも、近い役割の代替候補があるかを確認しろ。

## 要件A6｜採用前の評価前提
見つけたものをそのまま採用候補扱いにするな。  
フェーズ1では情報基盤として整理まで行い、採用可否は後続フェーズに回す前提を守れ。

## 要件A7｜継続調査可能性
後から再調査・深掘り・比較がしやすいよう、情報源と初期要約を必ず残せ。

# 【このフェーズで必ず守るべき探索方針】
## 方針P1｜公式を先に見る
まず公式 docs / release notes / 公式 GitHub を見る。  
その後に OSS / 公開エージェントへ広げる。

## 方針P2｜既知候補から出発しつつ広げる
既知候補は出発点として使ってよい。  
ただし、既知候補で探索を止めるな。

## 方針P3｜名前検索だけで終わるな
名称一致だけで満足するな。  
役割類似、用途類似、構成類似の候補も探せ。

## 方針P4｜全部自作前提を捨てる
最初から自作する前提で情報整理するな。  
既存資産流用の可能性を常に開けておけ。

## 方針P5｜採用判断はまだしない
フェーズ1は基盤構築である。  
この段階で採用・不採用を確定しすぎるな。  
後続フェーズのために比較可能な情報を揃えることを優先しろ。

## 方針P6｜更新前提で作る
初回収集で終わる構造にするな。  
あとから更新差分が追加できる構造にしろ。
