# 比較単位・比較マトリクス方針（Comparison Units & Matrix Plan）

> **担当**: グループ1  
> **ステップ対応**: フェーズ2作業指示書 ステップ3（比較単位の確定）  
> **最終更新**: 2026-04-12（グループ1）

---

## この文書の役割

本文書は、フェーズ2で候補を比較する際の**比較単位**（何と何を比べるか）と**比較マトリクスの設計方針**（どう並べて比べるか）を固定する。後続グループはこの枠組みに従って比較評価を実施すること。

---

## 比較の基本原則

1. **役割が違うものを一括比較しない**: routerとorchestratorを同じ表で比較しない
2. **比較単位を雑に混ぜない**: 同役割比較と公式vsOSS比較は別の観点として分ける
3. **フェーズ2支援部品も比較対象に含める**: 最終成果物向けだけでなく、今回の作業を支援する部品も評価する
4. **後続グループでGitHub再探索結果を追加できる構造にする**: 比較表に行を追加するだけで新候補を組み込める設計にする
5. **比較漏れを防ぐ仕組みを持つ**: 候補カタログの全候補が、いずれかの比較グループに所属していることを確認する

## パート1｜比較単位の定義

### 比較単位U1｜同役割比較

**目的**: 同じ機能的役割を持つ候補同士を、同じ観点で比較する

**やること**: 下記の比較グループごとに、`01_evaluation_criteria.md`の10軸で各候補を並べて評価する

**注意**: 異なる比較グループの候補を横断比較しない。ただし、接点がある場合（例: mcp-agentはorchestration兼MCPフレームワーク）はnotesに記録する

---

### 比較単位U2｜公式 vs OSS 比較

**目的**: 公式機能だけで十分か、OSS導入で価値が上がるかを判定する

**やること**: 各機能領域について、以下を比較する
- Anthropic公式機能のみで実現できる範囲
- OSSを追加導入することで得られる追加価値
- OSSの導入コスト・保守リスクと、公式機能の制約のトレードオフ

**比較の出し方**:
- 「公式のみで十分」→ OSS不要
- 「公式 + OSS補完が最適」→ 部分的にOSSを採用
- 「OSSの方が適合する」→ 公式の代替としてOSSを検討

---

### 比較単位U3｜自作 vs 流用 比較

**目的**: 既存候補を流用すべきか、Claude Agent SDK等を使って自作した方がよいかを判定する

**やること**: 各機能領域について、以下を比較する
- 既存候補をそのまま使う場合のメリット・デメリット
- Claude Agent SDK等で自作する場合のメリット・デメリット
- 一部流用 + 自作補完のハイブリッド案

**判定基準**: `01_evaluation_criteria.md` の E4（流用可能性）を主軸とし、E5（導入難易度）・E6（保守性）・E8（拡張性）を副軸として判定する

**判定結果**: B1〜B5（全面流用/部分流用/参考流用/自作優先/判断保留）のいずれかに分類する

---

### 比較単位U4｜今やる vs 後でやる 比較

**目的**: 価値はあるが今ではない候補を切り分ける

**やること**: 各候補について、E10（今やる意味）を主軸として以下を判定する
- 今触らないと後続フェーズがブロックされるか
- 今触っても仕様変更で手戻りするリスクがあるか
- 学習コストの前倒しに明確なメリットがあるか

**判定結果**: P1〜P5（最優先/高優先/中優先/低優先/凍結）のいずれかに分類する

## パート3｜比較マトリクスの設計方針

### マトリクスの構造

各比較グループ（CG-01〜CG-06）ごとに1つの比較マトリクスを作成する。

| 要素 | 配置 |
|------|------|
| **縦軸（行）** | 候補（item_id + 候補名） |
| **横軸（列）** | 評価軸（E1〜E10のうち、その比較グループに最も関連する軸を選択）+ compared_with + decision_state + build_vs_buy_state + reason_summary |
| **セル** | スコア（1〜5）+ 根拠の簡潔な記述 |

### 横軸の選択方針

全10軸を全比較グループに適用すると冗長になるため、各比較グループの性質に応じて**主要軸**（必須）と**副軸**（必要に応じて記録）を分ける。

| 比較グループ | 主要軸（必須） | 副軸（任意） |
|-------------|--------------|-------------|
| CG-01 router/gateway | E1,E3,E4,E5,E6,E10 | E2,E7,E8,E9 |
| CG-02 orchestration | E1,E2,E4,E5,E6,E8 | E3,E7,E9,E10 |
| CG-03 evaluation | E1,E3,E5,E6,E7,E10 | E2,E4,E8,E9 |
| CG-04 知識活用 | E1,E4,E8,E10 | E2,E3,E5,E6,E7,E9 |
| CG-05 実行エージェント | E1,E2,E3,E5,E7,E8 | E4,E6,E9,E10 |
| CG-06 支援部品 | E1,E3,E5,E10 | E2,E4,E6,E7,E8,E9 |

### 候補の追加方法

1. グループ2のGitHub再探索で新候補が発見された場合、該当する比較グループの表に行を追加する
2. 新候補が既存の比較グループに収まらない場合、新しい比較グループ（CG-07以降）を作成する
3. 追加時は必ず `phase1_information_foundation/02_candidate_catalog.md` にも登録し、item_idを付与する

### 比較漏れの防止方法

グループ3の評価記録作成時に、以下のチェックを行う:

1. `02_candidate_catalog.md` の全候補（F-001〜F-031 + 追加分）が、いずれかの比較グループに所属しているか確認する
2. どの比較グループにも属さない候補がある場合、「独立評価」として個別に評価記録を作成する
3. 複数の比較グループにまたがる候補（例: LiteLLMはCG-01とCG-06の両方）は、主たる比較グループで評価し、副次的な比較グループではnotesに参照記録を残す
